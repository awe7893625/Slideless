3. Pain Point Analysis 痛點分析

按嚴重度排序：Critical → High → Medium → Low

Critical

任務派發全靠人工指令，無事件驅動

影響：Rain 必須在線下指令，爬蟲/發文/監控/報告才會執行。Rain 不在 = 團隊停擺。
數據：Kai 看板 88 任務中 22 個 failed（25%），12 個永遠 pending。主因是沒有自動重試和排程機制。
解法：n8n 作為事件中樞，Webhook + Cron 驅動常規任務。

Critical

日報/週報從未穩定執行過

影響：lesson.md 記錄「5 區塊日報太重，沒人執行」（L-2026-03-06）。Rain 無法掌握每日進度。
數據：日報 SOP 存在但執行率接近 0%。三問制改進後仍不穩定。
解法：n8n 每日 21:00 Cron 自動拉數據 → 生成報告 → Kai Email 寄出。

High

爬蟲/發文零排程，全靠手動觸發

影響：TheOrder 商品價格/庫存可能過時數週。Buzz 社群發文靠 Rain 指派才動。
現況：Spider/Lynx/Nautilus 有能力但沒有排程引擎。Buzz 有帳號但沒有內容排程。
解法：n8n 排程驅動爬蟲 pipeline，結果自動寫入 DB + 通知。Buzz 發文排程佇列。

High

無統一檔案/圖片儲存層

影響：圖片散落 Supabase Storage / 本機 / Vercel；報告在 Slideless / /tmp / Email。無法統一管理、備份、CDN 加速。
解法：Cloudflare R2 作為統一儲存層（S3 API 相容），免費 10 GB/月。

Medium

GA4/GSC 數據查詢費時，非即時

影響：Atlas 查數據需要手動跑 API 腳本，Rain 想看流量要等。中控台 15 分鐘快取但需手動刷新。
解法：GA4 MCP Server 讓 Atlas 在 Claude Code 內即時查詢。n8n 排程每日自動拉數據存 cache。

Medium

無短網址 / 無 UTM 追蹤 / 無行銷連結管理

影響：EchoMarketer 做 SEO 策略但無法追蹤連結效果。社群發文連結無法追蹤轉換。
解法：Cloudflare Worker + KV 短網址服務，自動附加 UTM 參數。每日 10 萬次免費。

Medium

網站監控靠 shell script + LaunchAgent，不夠即時

影響：每日 12:00/21:00 才巡檢，中間 9-12 小時的停機無法發現。Uptime Kuma 在線但 alert 機制脆弱。
解法：Vercel Deploy Hook → n8n Webhook → 部署後自動健康檢查 + Discord 告警。R2 作為健康報告歸檔。

Low

GooayeAI 投資 pipeline 偶爾需手動重啟

影響：相對輕微，pipeline 已經 semi-auto。偶爾 RSS 抓取失敗需人介入。
解法：n8n 加入健康監控 + 自動重啟邏輯。低優先。

核心設計原則（from Darrell EP20）

Workflow > Agent

能用確定性工作流解決的事，絕不用 AI Agent。Agent 耗 3-6x token。日報生成、健康檢查、爬蟲排程都是確定性流程，應該用 n8n workflow。

Observability First

n8n 的核心優勢是每一步可視化。所有自動化工作流都能在 n8n UI 看到執行狀態、輸入輸出、錯誤點。取代 shell script 的黑箱執行。

6. Scope 範圍（P0 → P3）

P0 立即執行（Week 1-2）

6.1 n8n 基礎建設 — Zeabur 部署

負責：Forge 成本：$5/月 耗時：2-3 小時

做什麼：在 Zeabur 部署 n8n（Docker），設定持久化儲存、環境變數、域名（n8n.blankspaces.uk 或類似）。
預期效果：團隊有一個 24/7 運行的自動化引擎，所有後續工作流都建在上面。
關鍵配置：webhook URL、Kai API 連線、Discord webhook、Email SMTP。

6.2 日報自動化工作流

負責：Max + Forge 成本：$0 耗時：4-6 小時

做什麼：n8n Cron 每日 21:00 TST → 拉 Kai 看板 API（今日完成/失敗/pending）→ 拉 website-health-latest.json → 拉 GA4 API 日數據 → 用 Structured Output Parser 組合報告 → Kai Email 寄出。
預期效果：Rain 每天 21:00 自動收到日報 Email，不需要任何人工觸發。執行率從 ~0% → 95%+。
關鍵設計：用 n8n 的 HTTP Request node 直接呼叫 Kai API，不經過 Claude Agent（省 token）。失敗 → Discord @Rain 告警。

6.3 部署後自動健康檢查

負責：Forge + Keeper 成本：$0 耗時：3-4 小時

做什麼：Vercel Deploy Hook → n8n Webhook → 自動 ping 5 個網站所有健康端點 → 結果寫入 cache JSON → 異常 → Discord 即時告警。
預期效果：每次部署後 30 秒內完成健康檢查。停機偵測從 12 小時 → 5 分鐘以內。
增強現有：不取代 LaunchAgent 定時巡檢，而是在每次部署後「加一層」即時驗證。

P1 高價值（Week 2-4）

6.4 Cloudflare R2 統一儲存層

負責：Forge + Max 成本：$0（免費 10 GB/月） 耗時：4-5 小時

做什麼：建立 Cloudflare R2 bucket（rain-ai-storage）→ 寫 Python 上傳工具（S3 API）→ 爬蟲圖片、報告 PDF、健康檢查歸檔統一存 R2 → CDN URL 取代本機路徑。
預期效果：所有新產生的檔案走統一路徑。備份安全、CDN 加速、任何 Agent 可存取。
n8n 整合：n8n S3 Node 直接操作 R2（API 相容），工作流可直接上傳/下載。

6.5 Cloudflare Worker 短網址 + UTM 追蹤

負責：Rex + Forge 成本：$0（免費 10 萬次/天） 耗時：3-4 小時

做什麼：部署 Cloudflare Worker + KV → 短網址 API（go.blankspaces.uk/xxx）→ 自動附加 UTM 參數（source/medium/campaign）→ 點擊統計寫入 KV。
預期效果：Buzz 發社群文時自動生成追蹤短連結。EchoMarketer 可看連結效果數據。行銷可衡量。
管理介面：Kai 中控台新增「連結管理」頁面（John 前端）。

6.6 爬蟲排程自動化

負責：Max + Spider/Lynx（OpenClaw） 成本：$0 耗時：6-8 小時

做什麼：n8n Cron → 週二/週五觸發 → HTTP Request 呼叫 Kai API 建立爬蟲任務 → Spider/Lynx 執行 → 結果寫入 TheOrder DB → 價格/庫存變動 → Discord 通知。
預期效果：TheOrder 商品資料每週自動更新 2 次。Rain 只處理「價格異常波動」的例外通知。
關鍵細節：n8n 只負責「排程 + 建任務」，實際爬蟲仍由 Spider/Lynx 的現有代碼執行（Workflow > Agent 原則）。

6.7 GA4 MCP Server

負責：Atlas + Max 成本：$0 耗時：3-4 小時

做什麼：用 FastMCP Python 建 GA4 MCP Server → 註冊到 Claude Code → Atlas 可在 session 內直接查詢 GA4 數據（流量、事件、轉換）。
預期效果：Rain 問「今天流量？」→ Atlas 即時回答，不用手動跑腳本。
已有基礎：SA 已有 GA4 Viewer 權限（5 個 property），API 已啟用，Python 環境就緒。

P2 效率提升（Week 3-5）

6.8 Buzz 發文排程自動化

負責：Buzz（OpenClaw）+ n8n 成本：$0 耗時：5-6 小時

做什麼：建立發文佇列表（Kai DB 或 R2 JSON）→ n8n 排程每天指定時段觸發 → 從佇列取內容 → OpenClaw Buzz 執行發文 → 自動附加短連結 + 圖片。
預期效果：Rain/Noel 批次排好一週內容，自動依排程發出。免除每次手動派發。
前置：依賴 P1 短網址（6.5）。

6.9 任務失敗自動重試 + 告警升級

負責：Max + Kai 成本：$0 耗時：3-4 小時

做什麼：Kai Task API 加入 Webhook 機制 → 任務 failed → n8n 收到 webhook → 判斷重試策略（max 2 次）→ 仍失敗 → Discord @Rain + Email 告警。
預期效果：任務失敗率從 25% 降到 ≤10%（transient error 自動恢復）。Rain 只收到「真正需要人介入」的告警。
數據：目前 22 個 failed 任務中估計 40% 是暫時性錯誤（API timeout、lock 競爭）。

6.10 週報自動生成

負責：Noel + n8n 成本：$0 耗時：3-4 小時

做什麼：n8n 每週一 09:00 Cron → 聚合過去 7 天日報數據 → 統計高頻錯誤 / Agent 任務量 / 成功率 → 用 Structured Output Parser 生成結構化週報 → Deck 格式 → Email URL。
預期效果：Noel 週報從「應該做但沒做」變成「自動生成 + Noel 審閱潤飾」。
前置：依賴 P0 日報自動化（6.2）穩定運行 1 週以上。

6.11 MCP Inspector 本地測試環境

預期效果：新 MCP Server 開發效率提升，上線前可本地驗證。

6.12 n8n 工作流監控儀表板整合

負責：John + Kai 成本：$0 耗時：4-5 小時

做什麼：Kai 中控台新增 n8n 工作流狀態頁面（n8n API → 執行歷史 / 錯誤 / 排程狀態）。
預期效果：Rain 在中控台一站式看到所有自動化工作流的執行狀態。Observability 完整覆蓋。

6.13 GooayeAI Pipeline 遷移至 n8n

負責：Max 成本：$0 耗時：4-6 小時

做什麼：將 gooaye_pipeline.py 的 RSS → Vocus/Whisper → Gemini → Telegram 流程遷移到 n8n workflow，保留 gooaye_bot.py 的 Telegram 介面。
預期效果：pipeline 出錯時 n8n 自動重試 + 可視化除錯，不再需要手動重啟。

7. 不做什麼（排除清單）

  • 不建新的 AI Agent — 團隊已有 29 個 Agent，問題不是人不夠，是自動化不夠。本次只加「調度層」不加「執行層」。
  • 不遷移現有 Supabase 到其他 DB — Supabase 免費層足夠（已有 keepalive），遷移 ROI 不合理。
  • 不建自訂 Dashboard 取代 n8n UI — n8n 自帶的工作流 UI 足夠好，不需要額外開發。Kai 中控台只做「狀態嵌入」。
  • 不做 LINE Bot 整合 — 雖然 n8n 有 LINE Community Node，但團隊通訊走 Discord + Telegram，LINE 是客戶端（等專案需要時再做）。
  • 不換 Claude Code Max 方案 — $200/月已是固定成本，n8n 的意義是「減少不必要的 Agent 呼叫」而非換模型。
  • 不做 n8n 高可用（HA）架構 — 單實例 Zeabur 足夠。流量不大，不需要 worker + queue 分離。
  • 不遷移 Vercel 到 Cloudflare Pages — Vercel 已有多站部署，遷移成本高且沒有明確收益。
  • 不做 AI Agent 自動決策 — n8n 工作流是確定性流程（workflow），不讓 AI 在中間自行決策。AI 只在端點執行（Claude Code Agent）。遵循 Workflow > Agent 原則。

9. Risk Flags for Ray Phase 3 審查

以下是設計層面的風險，需要 Ray 在 Phase 3 做架構簽核時逐項審查。

R1: n8n 單點故障 — Zeabur 掛了所有自動化停擺

n8n 是所有自動化的中樞。如果 Zeabur 停機，日報/爬蟲排程/健康檢查/告警全部失效。目前設計是單實例無 HA。

緩解：LaunchAgent 保留為 fallback（不刪現有 shell script），n8n 只是「增強層」。n8n 本身加入 Uptime Kuma 監控。Zeabur SLA 99.9%。

R2: Webhook 安全 — n8n webhook URL 暴露可能被惡意觸發

n8n webhook 是公開 URL，任何人知道 URL 都可以觸發工作流。惡意觸發可能導致垃圾任務、API 額度浪費。

緩解：所有 webhook 加入 Header Token 驗證（n8n 支援 Header Auth）。Vercel Deploy Hook 用 secret。限制 webhook 來源 IP（Cloudflare、Vercel IP range）。

R3: Cloudflare R2 存取權限 — API Key 洩露 = 所有檔案被刪

R2 API Key 有完整讀寫刪權限。如果洩露在 git commit 或 env 檔中，攻擊者可刪除所有儲存的檔案。

緩解：API Key 只存在 n8n 環境變數 + Kai .env（不進 git）。Knox vault 登記。Sentinel hook 掃描。R2 開啟版本管理（可恢復刪除檔案）。

R4: n8n → Kai API 的認證 — 中控台目前無 API 認證

Kai 中控台 API（127.0.0.1:5002）目前沒有認證機制，任何能存取 localhost 的進程都可以呼叫。當 n8n 在外部（Zeabur）時，需要暴露 Kai API 到公網或透過 tunnel。

緩解：方案 A（推薦）：Cloudflare Tunnel 將 Kai API 暴露到 n8n，加 Bearer Token 認證。方案 B：n8n 改為本地部署（Docker on Mac），免去公網暴露。Ray 需決定走哪個方案。

R5: 爬蟲自動排程可能對目標網站產生壓力

自動化排程如果設錯間隔（例如每小時而非每週兩次），可能對 Rakuten 等目標網站產生過大壓力，導致 IP 被封或法律問題。

緩解：n8n Cron 設定需經 Ray 審查。最小間隔 = 12 小時。加入 rate limiter。爬蟲腳本已有 Rakuten WAF 規避邏輯。

R6: Workflow > Agent 邊界模糊 — 什麼該放 n8n、什麼留給 Claude

如果邊界不清楚，可能出現「n8n 也做 AI 決策」或「Agent 做應該是 workflow 的事」，造成 token 浪費和維護困難。

緩解：硬性規則 — n8n 只做確定性流程（HTTP call、data transform、conditional route、schedule）。任何需要「理解語意」或「生成內容」的步驟留給 Claude Agent。邊界寫入 CLAUDE.md。

R7: 遷移期間兩套系統並行 — LaunchAgent + n8n 可能重複執行

P0 上線後到 P1 穩定前，LaunchAgent（健康檢查、keepalive）和 n8n 會同時存在。可能出現重複觸發（兩份日報、兩次健康檢查）。

緩解：嚴格分階段切換。n8n workflow 驗證穩定（連續 3 天正常）後才停用對應的 LaunchAgent。並行期間用 flag 標記來源（launchagent vs n8n），去重。

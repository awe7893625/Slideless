03 · 入庫流程

法條從哪裡來？怎麼建向量？

法條資料的生命週期：法規全書 → 爬蟲下載 → 切成 chunk → OpenAI embedding → SQLite 向量庫。

1

爬蟲抓取

呼叫 POST /api/tax/crawler/run，後台觸發爬蟲到全國法規資料庫（law.moj.gov.tw）下載法條全文。每條法規儲存到 tax_laws 表，包含：law_name、article_no、content、doc_type、tax_type、topic_tags、source_url。

# 目前入庫的主要法規來源
來源: law.moj.gov.tw（全國法規資料庫）
格式: 每條文 → 一筆 tax_laws 記錄
表: m4.db → tax_laws（2,443 筆）
2

切塊（Chunking）

長條文切成語意完整的 chunk，每塊存到 tax_chunks 表。目前大部分法條是一條文對應一個 chunk。chunk 記錄：article_no、content、law_id（關聯 tax_laws）、embedding_id（向量庫 ID）。

tax_chunks: 2,541 筆
已嵌入: 2,537 筆（embedding_id IS NOT NULL）
待嵌入: 4 筆（空內容條文，可忽略）
3

向量化（Embedding）

呼叫 POST /api/tax/embed，批量對 pending chunks 呼叫 OpenAI text-embedding-3-small（1536 維）API，將法條文字轉成語意向量。向量以 BLOB 格式儲存在 SQLite。

# 向量文件格式（新格式）
document: "法規名稱 條號 topic_tags\n條文內容"
metadata: {law_name, article_no, tax_type, status: 'active', ...}
embedding: cosine similarity 用 blob（1536 維）

# 舊格式（部分條文）
document: "條文內容"（無法規名稱前綴）
metadata: {law_name, article_no}（無 status/tax_type）
4

入庫完成

向量庫（vectorstore/vectors.db → tax_laws 表）共 2,540 筆。2,161 筆有完整 embedding，379 筆是舊格式（無向量，靠關鍵字搜尋）。embedding_id 回寫到 tax_chunks 標記為已完成。

04 · 問題處理流程

收到問題後的 7 個步驟

從 Telegram 收到訊息到回傳答案，整個流程平均需要 30–60 秒（主要是 claude -p 的推理時間）。

1

接收訊息（tax_bot.py）

Telegram Bot 收到訊息後驗證 user_id。只有 Rain（7257793479）可使用。收到問題後呼叫 Kai API：

POST http://127.0.0.1:5002/api/tax/query
{
  "question": "韓商台灣分公司員工的租屋補貼...",
  "model_tier": "standard"
}
2

事實萃取（_extract_facts）

從問題中提取關鍵事實，判斷稅務類型（tax_type）用來過濾搜尋範圍。例如：「韓商台灣分公司 → 外籍人士優惠、所得稅」。

tax_type_filter: ['所得稅', '外籍人士優惠', 'OBU']
3

向量搜尋（_vector_search）

用問題的語意向量，在 2,540 筆法條 chunk 中找最相似的 20 筆。使用 cosine similarity。若 OpenAI API 可用 → 語意搜尋；否則 → 關鍵字 fallback（TF 加權）。過濾 status != 'active' 和已刪除條文。

# 搜尋流程
query_text → OpenAI text-embedding-3-small
→ cosine_similarity(query_emb, all_stored_embs)
→ top 20 by similarity score
→ filter: status == 'active'
→ filter: content != '（刪除）'
→ apply tax_type_filter

# Fallback（無向量時）：關鍵字匹配
relevance = 0.5 + 0.3 × (matched_keywords / total_keywords)
4

重排序（_rerank）

對向量搜尋的 top-20 結果再做精細排序，選出最相關的 8 筆送給 LLM。排序依據：doc_type（母法 > 解釋令）、score、關鍵字密度。

top-20 → rerank → top-8 送 LLM
5

LLM 推理（_reason → claude -p）

將 top-8 法條 + 問題組成 prompt，呼叫 claude -p（Claude Code MAX 訂閱，不消耗 API token）。claw-proxy 為 fallback（25 秒超時）。

# 模型優先順序
1. claude -p（Claude Code 訂閱）→ 無 token 費用
2. claw-proxy（fallback, timeout=25s）
3. 格式化 raw chunks 回傳（最後 fallback）

# Prompt 送入法條格式
【法源 1】所得稅法 第 4 條
{條文內容...}
來源：https://law.moj.gov.tw/...
6

法源驗證（_validate_legal_basis）

用 Regex 檢查 LLM 回答是否含有明確法源。若 LLM 自己說「查無明確法源」→ 不直接拒絕，改以 confidence: low 回傳完整回答內容。

# 合格法源格式（任一即可）
第X條、台財稅字第XXXXXXX號、法規名稱

# 驗證失敗 → confidence: low 仍回傳
confidence: "high" | "medium" | "low"
7

格式化回答（tax_bot.py format_reply）

Telegram Bot 收到 API 回應後，格式化為 Markdown。高/中信心度：顯示完整結構化答案（結論 + 法源 + 邏輯 + 會計處理 + 文件 + 風險）。低信心度：顯示 LLM 完整回答 + 免責聲明。

05 · 顧問定義

T-TAX 是誰？怎麼回答問題？

LLM 在 prompt 中被賦予「T-TAX」的角色定義，包含專業範圍、答題規則與回答格式。這是決定回答品質的核心設計。

角色身份

你是 T-TAX，台灣資深財稅顧問。

專業領域（全部都懂）
  • 所得稅法（個人 / 營利事業）
  • 營業稅法
  • 遺產及贈與稅法
  • 證券交易稅
  • OBU / DBU 境外金融業務
  • 銀行業會計實務
  • 外國企業台灣分支機構稅務（含韓商銀行）
回答規則（強制遵守）
  • 只能用提供的法條回答，不引用清單外的法規
  • 法條優先順序：母法 > 施行細則 > 解釋令（字號）> 函釋 > 問答集
  • 必須以「結論：」開頭，直接給出是/否/有條件結論
  • 找不到法源 → 直接說「查無明確法源，建議向國稅局函詢」
  • 信心度：有解釋令字號 → 高；只有母法 → 中；無明確法條 → 低
  • 精確、直接，不繞彎子
強制回答格式（每段必須有）
結論： 直接回答是/否/有條件，一句話
法源依據： 引用哪條法規，包含條號或函釋字號
判斷邏輯： 為何得出此結論（150字內）
會計與申報處理： 如何記帳、是否需扣繳、是否需申報
應備文件： 需要哪些憑證 / 文件
風險提醒： 可能被否准或補稅的情況

✅

T-TAX 會說「有法源」的條件

LLM 在提供的 8 筆法條中找到明確引用點，並且回答包含「第 X 條」或「台財稅字第 XXXXXXX 號」格式的法源。這時 confidence = high 或 medium。

⚠️

T-TAX 會說「查無明確法源」的條件

在提供的 8 筆法條裡找不到能回答該問題的明確條文。這時 T-TAX 遵守規則直說查無，confidence = low，但仍顯示模型的完整推理作為參考。

06 · 檢索機制

向量搜尋 vs 關鍵字搜尋

系統有兩套搜尋策略，依 OpenAI API 是否可用自動切換。向量搜尋理解語意，關鍵字搜尋靠詞頻。

項目向量搜尋（主要）關鍵字搜尋（Fallback）
觸發條件 OpenAI API 可用（有 OPENAI_API_KEY） API 不可用、或向量庫回傳空結果
原理 問題轉成 1536 維向量，對全庫 cosine similarity 把問題拆成關鍵字，計算每個法條命中的關鍵字數
優點 理解語意，「租屋補貼」也能找到「房屋租金」 不需要 API，快速
缺點 需要 OpenAI 費用（每次約 $0.0001） 舊格式法條文件沒法規名稱 → 法規名精確查詢失效
目前狀態 有效（OPENAI_API_KEY 已設定） 有效，但舊格式條文只靠內容比對

⚠️ 第9條 在向量庫，但格式是舊版

外國專業人才延攬及僱用法第9條（就業金卡規定）確實存在向量庫中（chunk id: e2fe4d25），有 embedding 向量。但文件格式是舊版：沒有法規名稱前綴、沒有 topic_tags。關鍵字搜尋時，搜「外國專業人才延攬及僱用法」找不到它；向量搜尋則正常（靠語意相似）。

❌ 第9條內容是就業金卡，不是獎學金免稅

外國專業人才延攬及僱用法第9條規定的是「就業金卡（Employment Gold Card）」的申請程序，與子女獎學金免稅無關。獎學金免稅的法源是所得稅法第4條第8款（政府、國際機構為獎勵進修給與的獎學金免稅）。T-TAX 對第9條說「查無明確法源」是因為搜尋到的法條確實沒有涵蓋「獎學金免稅」的具體規定。

08 · 改善建議

已知問題與修復方向

以下是根據本次診斷提出的具體改善項目，按優先級排列。

🔴 P1 — 重建舊格式 chunks 的向量

379 筆舊格式 chunks 的文件沒有法規名稱前綴，導致關鍵字搜尋失效。需要重置這些 chunks 的 embedding_id，重新以新格式（法規名稱 + 條號 + topic_tags + 內容）建立向量。

🟡 P2 — 補充外籍專業人才租稅優惠法規

外國專業人才延攬及僱用法的租稅優惠條文（第21條以後涉及租稅的部分）與財政部相關解釋令應補充到向量庫。目前解釋令只有111筆，這類案例最需要有字號的函釋。

🟡 P3 — 讓 LLM 可用自身知識補充

目前 prompt 規則是「只能用提供的法條回答」。可以改為：在提供的法條之外，LLM 可以補充自身知識，但需標注「此為模型推論，非 RAG 法源」，提高問題覆蓋率。

🟢 P4 — 增加韓商銀行專屬解釋令

針對韓商在台灣分公司的特殊情境（OBU存款利息、員工福利費用認列），補充財政部針對外商銀行的具體函釋，提升這類問題的法源準確率。

🟢 P5 — 修復 /api/tax/embed 錯誤

目前 POST /api/tax/embed 回傳未知錯誤。4筆 pending chunks 內容為空，需要查清楚 build_embeddings() 在處理空內容時的例外錯誤，加入空值保護。

🟢 P6 — 查詢日誌分析

734次歷史查詢中，哪些問題「查無明確法源」比例最高？可以用這些失敗案例優先補充對應法規，形成數據驅動的法規入庫策略。

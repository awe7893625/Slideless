Current State — 21 Tasks Breakdown

當前團隊所有任務統一使用 Sonnet。以下按建議模型等級分類：

L1（Haiku）— 9 個任務

L2（Sonnet） — 5 個必保任務

複雜邏輯、多層推理、失敗成本高（必須 Sonnet）

  • complex-code — 新功能實作、>100 行邏輯複雜代碼
  • complex-debug — 多層追蹤、根因分析、edge case 處理
  • API-design — RESTful 契約、幂等性、錯誤處理設計
  • architecture — 系統設計、模組邊界、資料流設計

Klay Routing Implementation

設計原則

  • 最小改動原則：配置化，無系統重構
  • 漸進降級：失敗自動 fallback 到上一層
  • 必保清單：5 個任務強制 Sonnet，無例外

虛擬代碼

route_task(task_type, model_suggestion):
  # 檢查必保清單
  if task_type in MUST_SONNET:
    return Sonnet

  # 按建議等級嘗試
  if model_suggestion == "L0":
    try OpenRouter Free
    fallback Haiku
    fallback Sonnet
  elif model_suggestion == "L1":
    try Haiku
    fallback OpenRouter Free (if available)
    fallback Sonnet
  else:
    return Sonnet

  # 超時/錯誤恢復
  on error:
    log incident
    fallback to Sonnet
    alert monitoring dashboard

YAML 配置（範本）

# ~/.claude/config/model-routing.yaml
routing:
  L0_FREE:
    models: ["openrouter_free", "fallback: haiku"]
    tasks:
      - crawler
      - tagging
      - rag_preprocessing
      - search
      - template
      - string_format
      - doc_reading
    timeout_seconds: 30
    max_retries: 2

  L1_HAIKU:
    models: ["haiku", "fallback: openrouter_free"]
    tasks:
      - classification
      - extraction
      - simple_code
      - comparison
      - prd
      - requirements
      - seo_draft
      - content_rephrasing
      - summary
    timeout_seconds: 60
    max_retries: 2

  L2_SONNET:
    models: ["sonnet"]
    tasks:
      - complex_code
      - complex_debug
      - api_design
      - architecture
      - type_system
    timeout_seconds: 120
    max_retries: 1

  monitoring:
    failure_threshold: 0.02  # <2% failure rate target
    cost_tracking: enabled
    alert_channel: discord

Fallback Strategy & Resilience

三層 Fallback 架構

Tier 1: L0 失敗 → Haiku

L0 Free 模型因超時或 API 限制失敗時，自動升級至 Haiku。預期失敗率 <2%。

Tier 2: Haiku 失敗 → Sonnet

Haiku 無法處理（邏輯複雜度高於預期）時，升級至 Sonnet。預期失敗率 <1%。

Tier 3: 所有模型失敗 → 人工介入

連續失敗或超時，記錄 incident → Discord 通知 Rain → 人工判斷 / 降級任務。

故障恢復指標

指標 目標 說明
L0 失敗率 <2% 允許偶爾超時或額度限制
Haiku 失敗率 <1% Haiku 理論上覆蓋所有 L1 任務
整體恢復時間 <5s 超時後自動 fallback
人工介入頻率 <1/週 低於此值表示系統健康

監控與告警

  • 即時監控：Kai 儀表板 `/api/model-routing/stats` — 實時失敗率、成本、延遲
  • 每日摘要：Discord → #operations — L0/L1/L2 分布、今日成本、異常任務
  • 告警規則：失敗率 >5% → Red Alert；>2% → Yellow Alert

6-Phase Implementation Plan

漸進式部署策略，每個階段獨立驗證，降低風險。

階段 時間 任務數量 預期節省 驗收條件
Phase 1 今天（3/24） 7 個 L0 任務 30-35% 失敗率 <2%，驗證 3 個任務
Phase 2 本週（3/24-3/31） 9 個 L1 任務 45-50% 失敗率 <1%，監控 5 天
Phase 3 觀察期（1 週） — 50-55% 整體失敗率 <2%，無人工介入
Phase 4 視情況（4 月） Sonnet 細粒度 55-65% P0 任務仍 100% Sonnet，測試 subset
Phase 5 預留 新模型集成 提高空間 待評估
永不動 — 6 個 P0 任務 — Must-Keep-Sonnet 永遠保持

Phase 1（今天）實施步驟

  1. Kai 後端加載 model-routing.yaml 配置
  2. 部署 L0 路由邏輯：task_type in ["crawler", "tagging", ...] → OpenRouter Free
  3. 設置 fallback：失敗 → Haiku，記錄 incident
  4. 監控 3 個 crawler 任務，驗證成功率 >98%
  5. 如成功，發布到所有 L0 任務

Phase 2（本週）實施步驟

  1. 啟用 L1 Haiku 路由
  2. 測試 classification / extraction 任務
  3. 監控 5 天，確認失敗率 <1%
  4. 擴展到全部 9 個 L1 任務

Phase 3（觀察）

全面監控，驗證整體成本節省達成、無異常失敗。

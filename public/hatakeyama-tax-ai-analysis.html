05 — Core Design Pattern

兩階段判斷系統（最值得學的設計）

會計科目判定不是一股腦丟給 AI，而是用兩個階段分層處理：

交易明細進入
→
Stage 1
關鍵字辭典比對
→
命中？
→
Stage 2
Claude API 推理
→
信賴度篩選
→
登錄 freee

Stage 1：關鍵字辭典比對

  • 14 個科目類別，每類 100+ 關鍵字
  • 全形→半形、大小寫正規化
  • 日語、英語、片假名全支援
  • 特殊模式：金額門檻（會議費 vs 交際費）
  • 速度快、成本零、準確率高

Stage 2：Claude API 備援

  • 只處理 Stage 1 無法判定的交易
  • 提示可用的 14 個科目，要求 JSON 回答
  • 附帶信賴度：high/medium → 登錄
  • low → 跳過，交人工確認
  • 節省 API 成本、避免過度依賴

為什麼這個設計很成熟

  • 成本控制：能用規則解決的不浪費 API 呼叫
  • 速度優先：關鍵字比對是毫秒級，API 推理是秒級
  • 可解釋性：Stage 1 的判斷邏輯透明，出錯容易追蹤
  • 漸進擴充：新模式出現 → 先讓 Stage 2 處理 → 穩定後加入 Stage 1 辭典
06 — Safety Design

7 條排除規則 — 「不做什麼」的設計

自動分錄最可怕的不是漏做，是做了不該做的。畠山在系統中硬編碼 7 種排除情境：

#排除類型排除原因處理方式
1內容不明的借記卡交易只有「借記+數字」沒有店名跳過，人工確認
2借款償還公庫、融資、貸款等 — 非一般費用跳過，人工分類
3社會保險費與稅金年金、健保、扣繳稅 — 特殊處理跳過，人工處理
4薪資支付從 freee 人事 API 取得員工名單排除跳過（另有薪資系統）
5投資與資產運用證券帳戶交易 — 非營業費用跳過，人工判斷
6ATM 提款與餘額調整帳戶間轉帳 — 非費用另行處理為轉帳
7公共事業費無對應科目設定跳過，人工對應

Rain 團隊的對應設計

我們已參考這個思路，建立了 .claude/rules/exclusion-rules.md，涵蓋 20+ 條排除規則：真錢交易、DB migration、DNS 變更、API Key 刪除、個資操作、稅務申報提交等。

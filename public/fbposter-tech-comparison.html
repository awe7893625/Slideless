01
執行摘要

我們的 FBGroupPoster 是 Chrome Extension（MV3），直接在瀏覽器環境操作 Facebook DOM。 競品 SnapPost（原 行銷快手）是 Windows/macOS 桌面應用，透過獨立程序控制瀏覽器。 兩者核心目標相同，但架構選擇帶來截然不同的能力邊界與限制。 我們在帳號安全和客製化有明顯優勢；SnapPost 在規模化和群組資料庫佔先。

SnapPost — 逆向推斷（Confidence: medium）

桌面應用 + 獨立 Chromium 實例（推測）

  • 推測根據：桌面 .exe 應用 + v5.7.4 版本命名，符合 Electron + Playwright/Puppeteer 架構
  • Cookie 管理：明確提及「Cookie 管理」功能，可能用 Chrome Extension 擷取 cookie 後注入獨立瀏覽器實例
  • 多帳號：Deluxe 支援 100 個 FB 帳號，顯示有帳號 profile 隔離機制（Extension 架構天然無法做到）
  • 群組資料庫：7,000+ 預建社團 — 必然有後端服務提供群組清單，我們是純本地
  • 反偵測：明確提及「環境模擬（Environment Simulation）」，比我們的指紋遮罩更完整
  • 無法確認：是否直接呼叫 FB 非公開 API（如 FAP）；若是則穩定性遠高於 DOM 操作

自動刪文技術對比
我們的實作

chrome.alarms + DOM 操作

  • 排程由 chrome.alarms.create 觸發，精度 ±1 分鐘
  • 一般貼文：開啟 postUrl tab → 找 [aria-label*="動作"] 三點選單 → TreeWalker 找「刪除貼文」→ 確認 dialog
  • 待審貼文（無 URL）：導到 mbasic/groups/{id}/my_pending_content → 找第一個「刪除」連結
  • 退出前安全檢查：只在有明確 postUrl 格式（/posts/、/permalink/、story_fbid=）時才執行，防止刪錯文
  • 已知問題：confirm dialog 的「刪除」按鈕有時點擊無效（FB 某些版本 dialog button 無法 simulateClick）
  • 已知問題：手動刪文後，Extension 記錄仍顯示 pending_delete，需手動清除
SnapPost（行銷快手）

「設定貼文存續時間」每週清理

  • 使用者設定「貼文存活天數」和目標社團範圍
  • 每週自動掃描並刪除過期貼文
  • 桌面應用可保持背景常駐，不受 Service Worker 生命週期限制
  • 推測：以 Playwright headless 控制瀏覽器，DOM 操作更穩定
  • 優勢：UI 清晰（設定存活時間），使用者不需要記得手動刪
  • 優勢：週期性批次清理，比我們每筆獨立排程更有效率

09
改善建議（優先排序）
01

WebGL 指紋動態化

目前 getParameter 回傳固定的「Intel Iris OpenGL Engine」，所有用戶指紋相同。應建立一個顯卡字串池（實際市場常見顯卡型號，約 20-30 種），每次安裝時隨機選取並固定，或每 N 天輪替。這是當前防偵測最明顯的漏洞。

工時: ~2小時
02

刪文 Confirm Dialog 修復

目前 simulateClick 在某些版本的 FB confirm dialog 上無效，根本原因是 FB 的 button 使用 React 合成事件，dispatchEvent 不一定能觸發。解決方案：改用 pointerdown + pointerup + click 三連擊，或搭配 element.focus() + enter keydown 作為備援。

工時: ~3小時
03

群組預建資料庫（縮小與 SnapPost 的差距）

SnapPost 最大賣點之一是 7,000+ 預建高質量群組。我們可以建立「房仲專屬群組庫」：爬取台灣主要縣市的房屋買賣/租屋相關 Facebook 群組，建立結構化清單（群組 ID、名稱、成員數、地區、更新日期），打包進 Extension 或作為雲端 API 提供。這個差異化版本（「台灣房仲群組庫」）比 SnapPost 的通用庫更有針對性。

工時: ~2-3天（含爬蟲 + UI 整合）
04

chrome.storage 資料加密

SnapPost 使用 Argon2 加密帳號和授權資料，我們的 chrome.storage 是明文存儲。雖然 Chrome Extension 的 storage 已有 sandbox 保護，但對進階用戶（透過 Chrome DevTools 存取）是可見的。建議使用 AES-GCM 加密 license key 和帳號相關敏感資料，密鑰可從 device ID 派生。

工時: ~4小時
05

多 Chrome Profile 支援（帳號隔離）

目前我們共用使用者 Chrome Profile，所有操作都在同一個 FB 帳號下。SnapPost 支援 100 個帳號切換。Chrome Extension 無法直接切換 Profile，但可以：（1）在設定中讓使用者切換「目前操作帳號」標識，提示他們在正確 Profile 下使用；（2）長期可考慮提供 Chromium-embedded 的桌面版本解決此根本問題。

工時: 中長期規劃
06

www compose 選擇器穩定化

目前 clickComposeArea 和 clickPostButton 依賴文字節點（「留個言」、「發佈」）和 TreeWalker，FB 改版後容易失效。建議加入 MutationObserver 監控 dialog 開啟狀態，並建立選擇器版本管理機制（記錄哪個版本在哪個日期失效），快速定位並修復 regression。

工時: ~1天

競爭態勢總結
我們的優勢
  • 房仲垂直整合（591/HouseBox 爬蟲 + PDF 匯入）
  • 社團規則自動偵測（About 頁面解析）
  • 群組自訂模板（佔位符系統）
  • Spintax 巢狀引擎 + Round-Robin 輪替
  • 跨平台（任何 Chrome 環境）
  • Marketplace 發文整合
  • Smart Delay 自適應間隔
  • 完整的待審文處理流程
  • 開放原始碼可審計（安全性透明）
SnapPost 的優勢
  • 7,000+ 預建高品質群組庫
  • 多帳號支援（最多 100 個）
  • Argon2 加密本地資料
  • 帳號 Cookie 隔離
  • 背景常駐（不受 SW 生命週期限制）
  • 更完整的環境模擬反偵測
  • 週期性批次刪文（每週清理）
  • 一對一遠端教學服務（降低客服門檻）
  • 3,000+ 客戶的市場驗證

結論：我們在房仲垂直市場有明顯差異化優勢，SnapPost 在通用規模化行銷較強。 最高優先的改善項目是 WebGL 指紋動態化（2小時，高 ROI）和刪文 confirm dialog 修復（3小時，解決已知痛點）。 中期目標是建立台灣房仲專屬群組資料庫，這是同時防禦和擴大護城河的最佳策略。

問題背景：為什麼要做這個

04 · scope-card-gate.sh v2

Gate 重構：修復 4 個 Bug + SESSION_KEY 精度

速度減緩（speed bump），不是硬鎖（hard lock）。Tiny 任務零摩擦，30 分鐘窗口自動過期。

Bug 1 — v1 問題

Session 級 gate（永久解鎖）

一旦解鎖，整個 session 後續所有任務全部放行，等同沒有 gate。

Bug 1 — v2 修復

30 分鐘時間窗口（自動過期）

TIME_BUCKET = int(time()/1800) 每 30 分鐘自動過期，新窗口重新攔截一次。

Bug 2 — v1 問題

Stale gate file（跨 session 持久）

CLAUDE_SESSION_ID 空值 fallback "main"，/tmp 跨 session 持久化，舊 gate file 讓新 session 永遠跳過。

Bug 2 — v2 修復

Session + 時間戳複合 key

scope-{SESSION}-{TIME_BUCKET} 不同時間必然不同 key，stale 問題消除。

Bug 3 — v1 問題

Tiny 任務無豁免（全攔截）

所有 Edit/Write 強制 IS_WORK=1，改一行 config 也要 Scope Card，過度摩擦導致習慣性繞過。

Bug 3 — v2 修復

程式碼檔案才攔截，豁免清單

豁免 .sh .json .md .yaml .toml .env .plist .html 等，只攔截 .py .ts .tsx .jsx .js 等程式碼檔案。

Bug 4 — Ray Issue 1 + SESSION_KEY 精度

明文解鎖指令 + 分鐘級 fallback

block 訊息含解鎖路徑可自行執行；SESSION_KEY fallback 精度到分鐘，分鐘交界時重複觸發。

Bug 4 — v2 修復

移除解鎖指令 + 小時級 fallback

block 訊息不含解鎖路徑；SESSION_KEY fallback 改為 date +%Y%m%d%H（整小時穩定）。

✅ 新設計邏輯（v2）

第一次修改程式碼檔案 → block + 顯示 Scope Card 格式 + 自動建立 gate file → 後續 30 分鐘同窗口放行 → 下一個 30 分鐘窗口重新攔截一次。Claude 輸出 Scope Card 後重試即可繼續，無需手動解鎖指令。

06 · CLAUDE.md Core Rule #0

頂層強制規則：Core Rule #0 新增

放在 Core Rules 區塊第一條，前 100 token context 位置，最高視覺優先度。

CORE RULE #0 — DISPATCH ENFORCEMENT（Standard+，無例外）

⛔ Dispatch enforcement: 任何 Edit/Write 之前必須先輸出 Scope Card。Grade ≥ Standard → Klay 只規劃，不 implement，必須 dispatch。Grade = Major/Critical → 先 3-Phase（Scout → Ray）再 dispatch。違反 = session review Q4 必須說明，連續 3 次 → Klay audit。

為什麼是 Rule #0

原本 Scope Card 規則在 Agent Behavior 區塊第 5 項，距 context 開頭較遠。Rule #0 放在 Core Rules 第一條，確保在高度壓縮的 context 中仍然可見。

與 session-start.sh 的關係

session-start.sh 已有完整 R-00 提示（lines 57-76）。Ray 明確指出不要再加重複提示。Core Rule #0 是 CLAUDE.md 層面的規則加固，session-start 是運行時注入，兩者互補不重疊。

07 · Honest Assessment

誠實評估：這能解決問題嗎？

Ray 的核心觀點：防線假設 Claude 不看防線；實際上 Claude 讀了所有 context 再行動——規則清晰比技術卡關更有效。

✅ 短期效果（技術層）

scope-card-gate 現在正確攔截，不再被 stale gate file 繞過。30 分鐘窗口確保每個工作窗口至少強制確認一次 Scope Card。Tiny 豁免減少摩擦。

✅ 中期效果（行為層）

Core Rule #0 + session-start R-00 雙重覆蓋，提高 Scope Card 合規率。dispatch-reminder 4+ 檔案時要求說明，增加 accountability loop。夜班 dispatch_rate 統計持續可見。

⚠️ 長期限制（仍然存在）

Klay 本身是 Claude，speed bump ≠ hard lock。Fresh session 第一個行為仍靠 context。subagent context 中 hooks 是否生效待確認。Noel 夜班 session 行為待觀察。

🎯 設計選擇（已知，接受）

Hard lock 會阻礙合法的緊急修復（P0/P1）和 Tiny 任務，帶來過高摩擦成本。7.5/10 是在「有效強制」與「正常工作不受干擾」之間的最佳平衡點。

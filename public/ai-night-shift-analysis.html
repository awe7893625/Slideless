01 · 專案概覽
AI Night Shift 是什麼
TL;DR

AI Night Shift 是 JudyaiLab 開源的多 Agent 夜班排程框架。核心概念：用 cron 排程在非工作時間自動執行 AI Agent（Claude Code、Gemini、自訂 Agent），透過檔案式通訊（night_chat.md + JSON inbox）讓多個 Agent 協作，產出報告。

02 · 架構分析
核心機制
Execution Flow
Cron Trigger → Load config.env → PID Lock（防止重複執行） → Pre-Phase Plugins（健康檢查、備份） → [Round Loop × MAX_ROUNDS] ├ Timeout 檢查 ├ Rate-limit 偵測 ├ 載入 Prompt Template（{ROUND}, {DATE} 變數替換） ├ 讀取 night_chat.md + bot_inbox 上下文 ├ Adapter 執行 Agent ├ 產出 Round Report └ 寫入 night_chat.md → Post-Phase Plugins（git summary、通知） → 產出 Daily Summary → 釋放 PID Lock
通訊機制

檔案式多 Agent 通訊

  • night_chat.md — Append-only 共享日誌（[HH:MM] Agent: 訊息）
  • bot_inbox/ — 每個 Agent 獨立 JSON 佇列，處理完移到 done/
  • msg.sh / notify.sh — Agent 對 Agent 直傳訊息

優點：零外部依賴、簡單易部署。缺點：不支援分散式、無即時同步、無 schema 驗證。

Adapter Pattern

每個 CLI Agent 需實作 6 個函式

  • adapter_run() — 執行 Agent（非互動式）
  • adapter_build_flags() — 生成 CLI 參數
  • adapter_check_rate_limit() — 偵測 429/quota
  • adapter_check_completion() — 辨識完成訊號
  • adapter_verify() — 確認工具已安裝
  • adapter_name() — 顯示名稱

04 · 可借鏡的優勢
他們做對了什麼

夜班自動排程 — 我們的最大缺口

他們的核心價值：非工作時間讓 Agent 自動幹活。配合 cron + 時間窗口（WINDOW_HOURS）+ 輪次限制（MAX_ROUNDS），Agent 在深夜自動處理待辦事項、跑回歸測試、做代碼維護。我們目前的 Agent 全依賴 Rain 手動啟動。

Adapter 抽象層 — Agent 可插拔

統一介面讓不同 CLI Agent（Claude、Codex、Aider）用同一套排程系統。我們的 spawn-klay.py / spawn-ray.py 是硬編碼的，沒有統一 adapter。

Rate-Limit 自動處理

偵測 429 → 等 60 分鐘 → 重試。我們目前遇到 rate limit 就手動處理或放棄。

Plugin 三階段 Hook

Pre（健康檢查）→ Task（品質清理）→ Post（git summary + 報告），每個 plugin 5 分鐘 timeout、非阻塞。比我們的 Claude hooks 更有結構。

完成訊號機制

Agent 輸出 NIGHT_SHIFT_COMPLETE 表示完成；連續兩次 → 提前結束。簡單但有效。

05 · 他們的弱點
我們已經超越的地方

無審查機制

Agent 產出直接 commit，沒有 code review、沒有 veto。我們有 4 層審查。

單機限制

PID lock 假設所有 Agent 在同一台機器。我們的 OpenClaw 已支援遠端 Agent。

Bash-Only

核心全是 shell script，難以維護、測試、擴充。我們用 Python + structured API。

無 Schema 驗證

JSON 訊息無驗證，格式錯誤靜默失敗。我們的 Kai API 有 Pydantic schema。

粗糙的 Dashboard

靜態 HTML 需手動上傳檔案。我們的 Kai 前端即時更新。

無角色分工

所有 Agent 平等，無審查者/實作者/協調者區分。我們有完整的 chain of command。

06 · 優化方案
Rain AI Team 實施計畫
Phase 1 · 高優先 — 夜班排程系統

Kai Night Shift Mode

在 Kai 中控台新增夜班模式，每晚自動執行待處理任務。

  • 架構：LaunchAgent（cron）→ 呼叫 Kai API /api/night-shift/start
  • 執行者：Max（後端 API）+ Forge（LaunchAgent 部署）
  • 核心功能：
    • 自動從 task board 撈「pending」任務，按 priority 排序
    • 自動 dispatch Agent（借鏡 adapter pattern 但用 Python）
    • 時間窗口限制（23:00-06:00 TST，可設定）
    • Rate-limit 自動偵測 + 等待重試
    • 每輪產出結果寫回 task board
    • 晨間報告自動 email Rain
  • 優於原版：整合 Kai task board（非外部 Linear）、4 層審查不跳過、Python 非 Bash
  • 估計：Major 級任務
Phase 2 · 中優先 — Agent Adapter SDK

統一 Agent 呼叫介面

將 spawn-klay.py、spawn-ray.py 統一為 Agent Adapter SDK。

  • 統一介面：AgentAdapter.run(prompt, timeout, config)
  • 內建功能：rate-limit 偵測、completion signal、health check、telemetry
  • 支援：Claude Code / Codex / Gemini CLI / OpenClaw agents
  • 執行者：Max
Phase 3 · 低優先 — Plugin 系統升級

結構化 Hook 系統

將現有 Claude hooks 升級為三階段 plugin 架構（Pre/Task/Post），每個 plugin 有 timeout + 非阻塞執行。

  • Pre-phase：健康檢查、環境驗證、備份
  • Task-phase：品質門檻（lint score、test coverage）
  • Post-phase：自動 commit summary、Discord 通知、晨間報告
  • 執行者：Forge

09 · 結論
行動建議

結論：借鏡排程概念，但用我們的架構實現

AI Night Shift 的核心價值是「非工作時間自動執行 Agent」這個概念。但他們的實現（Bash + 檔案通訊 + 無審查）對我們來說太粗糙。

建議做法：

1. Phase 1 立即啟動：在 Kai 中控台實現 Night Shift Mode（Python，整合現有 task board + 審查機制）

2. Phase 2 排入 backlog：統一 Agent Adapter SDK，取代零散的 spawn-*.py

3. Phase 3 觀望：Plugin 系統升級，待 Phase 1 穩定後再做

預估效益：每日多 6-7 小時 Agent 產能（23:00-06:00），Rain 不在時系統仍在推進任務。

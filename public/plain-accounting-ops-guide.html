使用者只需要做一件事：選「你做了什麼」+ 填金額。系統在背後完成三件事：

1. 白話文 → 會計科目

10 個白話分類各對應一組借方/貸方科目。例如「收到客戶付款」= 借：銀行（111301）/ 貸：銷貨收入（411100）。對應表寫死在 api.py 的 PLAIN_CATEGORIES 字典裡。

2. 自動建立分錄

FastAPI 收到前端請求後，透過 Odoo 的 XML-RPC API 呼叫 account.move.create()，建立一筆含兩行 line_ids（借方 + 貸方）的日記簿分錄。

3. 交易對象管理

如果使用者填了「對方名稱」，系統先搜尋 Odoo 的 res.partner。找到就連結，沒找到就自動新建。下次記帳同一對象就不用重填。

技術架構

  • 前端：靜態 HTML（static/index.html）— 零框架，純 JS fetch API
  • API：FastAPI（api.py, :8070）— 白話文分類 → Odoo XML-RPC 代理
  • 後端：Odoo 17 Community（Docker, :8069）— 會計核心 + 台灣科目表
  • 資料庫：PostgreSQL 16（Docker）— Odoo 的唯一 DB
  • 會計科目：l10n_tw 模組 — 131 個台灣標準科目

03 — Data Integrity

入帳正確性保障機制

「系統會不會記錯帳？」— 有四道防線：

防線 1：科目硬編碼（最關鍵）

每個白話分類對應的借方/貸方科目是寫死在程式碼裡的，不是 AI 推理出來的。使用者選「交通費」，系統永遠記到 611400（旅費），不會因為任何原因變成其他科目。這跟畠山的 Stage 1（關鍵字辭典）是同一個設計思路。

防線 2：Odoo 複式簿記強制

Odoo 的 account.move 模型有內建驗證：每筆分錄的借方合計必須等於貸方合計，否則 Odoo 拒絕建立。這是 ERP 層級的硬限制，程式碼繞不過去。即使 API 傳入不平衡的金額，Odoo 會直接報錯。

防線 3：草稿 → 過帳兩階段

白話文記帳建立的分錄狀態是「草稿」(draft)。需要會計師在 Odoo 後台審閱後手動「過帳」(post)，才會正式生效。這意味著即使使用者選錯分類，會計師有機會在過帳前修正。

防線 4：審計軌跡

Odoo 記錄每一筆分錄的建立時間、建立者、修改歷程。所有操作都有完整的 audit trail。加上我們的白話文 API 在 ref 欄位標註了分類名稱（例如 [收到客戶付款] 3月網站建置費用），方便事後追溯。

跟畠山系統的差異

  • 畠山：兩階段判斷（關鍵字 → Claude API 推理），有 AI 介入
  • 我們：單階段硬對應（白話分類 → 科目），零 AI 介入
  • 我們更簡單但更安全：沒有 AI 推理 = 沒有推理錯誤的可能
  • 代價：使用者只能選預設的 10 個分類；超出範圍的要到 Odoo 後台手動記
  • 未來擴充：可以加 Stage 2（Claude API 推理），讓使用者打白話文自動判斷分類

05 — Cloudflare

Cloudflare 適合部署嗎？

直接部署 Odoo 在 Cloudflare？不行

Workers 記憶體上限 128MB，且不支援 Python 原生套件（lxml, psycopg2）。Pages 只能放靜態網站。Cloudflare 沒有 PostgreSQL 服務。Odoo 無法直接跑在 Cloudflare 上。

用 Cloudflare Tunnel？非常適合

Odoo 跑在你自己的電腦（或 VPS），透過 Cloudflare Tunnel 建立加密通道暴露到網路。免費、不用開 port、自動 SSL、有 DDoS 防護。目前最快的上線方式。

06 — Security & Backup

安全性與備份機制

防火牆

UFW 只開 22(SSH) + 80 + 443。Odoo 的 8069 和 8072 不對外暴露，只透過 Nginx 反代或 Cloudflare Tunnel 存取。

暴力破解防護

Fail2Ban 監控 Odoo 登入日誌。5 次失敗 → 封鎖 IP 1 小時。SSH 改為 key-only、禁 root 登入。

Odoo 硬化

list_db=False（禁止從網頁列出/建立 DB）、proxy_mode=True、強 admin 密碼。DB 只聽 localhost。

每日自動備份

凌晨 2:00 自動 pg_dump（DB）+ tar（filestore）。保留 30 天。上傳到 Cloudflare R2（免費 10GB）作為異地備份。

05 / Auto Audit Agent
後續建置規格（A 階段）

此 Inventory 作為 Auto Audit Agent 的基線 schema。Runtime 補充以下資料後，輸出定期 Deck 報告寄送 Rain。

INPUT/api/audit/run — 接受的資料來源
以 inventory.json 為 base schema，runtime 補充以下欄位：
1. 每個 agent 最近 N 次任務的 duration → 來源：Kai task board (GET /api/tasks?assigned_to={agent}) 2. 外部 agent 的 API call count + token usage → 來源：OpenClaw logs（需新增集中追蹤） 3. Forge CI pass rate → 來源：CI logs（Vercel deploy history） 4. Sentinel alert frequency → 來源：Kai task board (GET /api/tasks?created_by=sentinel)
OUTPUT/api/audit/report — 輸出格式
Nova 生成 Deck HTML → Slideless 部署 → Email Rain（asdapper8866@gmail.com）
報告包含： - Agent 健康分（per agent 健康指標） - Per-Call 成本趨勢圖（週/月） - HIGH 風險項目追蹤（R-01 ~ R-10 狀態） - Observability 缺口進度 - 新增/退役 Agent 變更記錄
TRIGGER自動執行條件
由 Noel 夜班模組觸發，或手動呼叫 /api/audit/run。
排程：每週一 09:00 TST（LaunchAgent 或 Kai cron） 手動：Klay dispatch → Noel 執行 → Nova 報告 更新觸發：任何 roster 變更後立即同步 inventory.json + 重跑審計
PRIORITY立即行動項目（依風險等級）
HIGH — 需優先處理： R-01 Noel fallback 機制（Gemini 掛掉時的替代方案） R-02 Ray fallback（codex-plus → OpenRouter → Vick 明確化） R-04 Nautilus token limit 設定（max_tokens 或 cost_limit per task） MEDIUM — 下階段處理： R-05 Kai task token 欄位區分訂閱/per-call R-06 Kai latency tracking（task duration 已有，需結構化展示） R-07 OpenClaw cost log 串接 Kai R-09 Nova Email href bug 確認修復狀態 LOW — 排進 backlog： R-08 補齊 Atlas/Pixel/Vick SKILL.md R-10 Ray fallback 成本估算文件化

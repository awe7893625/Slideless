架構問題識別（6 項）

🔴 P0
Dispatch 執行率嚴重不足
問題
CLAUDE.md 規定 Standard+ 任務必須 dispatch，Week 3+ 目標 ≥70%，實際僅 7%
影響
Klay 成為系統瓶頸；28 人團隊的投資幾乎浪費
根因
Session 中任務分類判斷仍由 Klay 自行處理，未落實 Scope Card 升級觸發 dispatch
建議
強化 dispatch hook（dispatch-reminder.sh）的攔截強度，從警告升為強制阻擋
🔴 P1
Spider/Lynx reports_to 資料錯誤
問題
team.json 中 Spider 和 Lynx 的 reports_to 仍指向 "milo"，但 Milo 已凍結（2026-03-13）
影響
爬蟲任務的指揮鏈斷裂，夜班 dispatch 可能路由失敗
修復
更新 team.json 中 Spider/Lynx 的 reports_to 為 "noel"，並同步 agent_task_service.py 的 TASK_ROUTING
🟡 P2
Investment Phase B 啟動條件未定義
問題
Fundamental/Technical/Macro Analyst + Bull/Bear Researcher（5 位）已加入 team.json，但「Phase B」何時啟動完全沒有文件
影響
5 位 Opus/Sonnet 等級 Agent 白白列在 roster，造成 roster 虛胖
建議
明確定義 Phase B 啟動條件（例如：Phase A 完成 10 份報告且 Rain 評估正確率 ≥70%）
🟡 P2
Knox vs Sentinel task_types 重疊
問題
Knox 的 task_types: ["security", "vault", "api_key"] 與 Sentinel 職責有重疊
澄清
Sentinel = 掃描/偵測（git hook 自動執行）；Knox = 密鑰管理/帳號生命週期（人工觸發）
建議
更新 Knox 的 task_types 為 ["vault", "account_lifecycle", "key_rotation"]，避免 router 混淆
🟡 P3
夜班任務重複建立
問題
任務板顯示同一個夜班任務（如「Ray 團隊架構週審」）有 3 個重複實例同時執行
根因
night-shift-prep.py 可能在多次執行時未檢查重複，或 auto_start: true 沒有 dedup 保護
建議
夜班任務建立前加 idempotency 檢查（依 title + date 去重）
🟢 P4
OM（Operations Manager）評估中
狀況
OM 於今日（2026-03-23）正式加入，task_types: ["coordination", "escalation"]，但 task board 尚無記錄
建議
2 週後（2026-04-06）評估 OM 的 09:00 作戰室摘要是否準確觸發；Quill 在第一次 OM 任務後做 lightweight review

人力配置評估

配置合理性

  • 核心工程鏈（Max→Rex→John→Iris→Forge→Scout→Ray→Quill）設計完整，但實際上幾乎所有工作都回流到 Klay
  • 根本問題：Dispatch 是人（Klay）的行為規範問題，不是架構問題。架構設計本身合理，但執行層的 dispatch 意願/機制不足

建議重點改善

  • 把 dispatch hook 從「提醒」升為「強制阻擋」（3+ 檔案修改 → 不填 dispatched_to → 拒絕執行）
  • 每週架構審查應包含 dispatch rate 趨勢圖（目標：Week 4 達 50%）
  • 對 Orion/Lab/Lens 定義「主動觸發日曆」（例如：Lab 每月第一週做模型 benchmark）

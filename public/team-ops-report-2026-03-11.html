Ray 架構審查

7 項發現 全修復

Ray（Codex Plus CLI）對 spawn-worker.py 進行架構審查，發現 3 HIGH · 1 MID-HIGH · 3 MID。點擊展開每項詳情。

HIGH-1
assigned_to 錯誤 + 重複偵測失效
Worker 任務錯誤歸屬 Klay 帳下
任務建立時 assigned_to="klay"，導致 Worker 任務出現在 Klay 名下，Dashboard 顯示錯誤。check_duplicate query 查的是 ?assigned_to=klay，Worker 任務完全無法觸發重複偵測。

修復：assigned_to="worker"，check_duplicate URL 參數同步更新。
HIGH-2
Shell Injection via Task Title（Ray PoC 驗證）
$(whoami) 注入成功 RCE
原本 title 直接插入 prompt 字串，LLM 生成 shell 指令。攻擊者在 title 放入 $(whoami) 即可 RCE。Ray 實際 PoC 驗證攻擊路徑成立。

修復：write_escalate_script() 預先生成 escalate_{task_id}.sh，所有參數用 shlex.quote() 包裹。LLM 只看到不含 title 的固定 script path。
HIGH-3
Closing Tag Bypass（兩個腳本都受影響）
</system> 可繞過 sanitize_for_prompt
sanitize_for_prompt() regex 只匹配 <system>，不匹配 </system>，攻擊者可用結尾 tag 注入 prompt 結構。spawn-klay.py 有相同漏洞。

修復：Regex 改為 <\s*/?\s*(system|assistant|user|task_input)\s*/?>，同時將 $(...) 命令替換和反引號加入 DANGEROUS_PATTERNS。兩個腳本同步修復。
MID-HIGH
Major 任務路由錯誤（Worker 自行執行）
Major 等級應轉 Klay 但由 Worker 直接跑
原本路由：Critical → Klay、Major → Worker 執行。Major 任務（3+ 檔案、前後端同時改）由 Worker 直接執行，缺乏 3-Phase Planning，風險過高。

修復：Major 和 Critical 均走 write_escalate_script() → spawn-klay.py。Worker 只處理 Standard 和 Tiny。
MID ×3
離線 Fallback · tmux 錯誤 · Noel 防幻覺
3 項穩定性強化
Kai 離線 Fallback：Kai 離線時原本 crash。現在寫入 offline_queue.jsonl，生成 local-{uuid} task ID，繼續執行不中斷。

tmux Return Code：tmux new-session 失敗無回應。加入 proc.wait() + returncode != 0 判斷，失敗時 fail Kai 任務並 exit 1。

Noel 防幻覺 +3 條：openclaw.json 新增規則，禁止捏造測試輸出、任務建立 ≠ Agent 啟動、通知聲稱需真實 API 回傳。

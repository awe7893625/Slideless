01 · BACKGROUND

任務背景：為什麼做 Observability？

Ray 企劃書指出三條 Tier 1 升級路線，Route B（Observability）ROI 最高，是最值得先做的一步

Ray 企劃書 — 三條升級路線

  • Route A — Graph Engine：DAG 任務依賴管理（複雜度高）
  • Route B — Observability：LangFuse 追蹤（ROI 最高，先做）
  • Route C — Cloud Scaling：多機器擴展（需要先 B）

Observability 解決什麼問題？

  • 每個 AI Agent 任務執行過程「黑盒子」— 不知道用了多少 token、耗時多久
  • 多個 subagent 協作時無法追蹤哪個環節最慢/最貴
  • 沒有歷史數據就無法做模型比較或 prompt 優化
  • 現在：每完成一個任務，Kai 自動 fire trace 到 LangFuse

Colima Docker Runtime

langfuse_service.py 用 stdlib urllib.request 直接打 REST API，零外部依賴，與 SDK 版本完全脫鉤。daemon thread 確保不阻塞 Kai API response。

05 · PROBLEMS & SOLUTIONS

遇到的問題：4 個障礙全部解決

每個問題都找到了更好的替代方案

Problem 01
Docker Desktop 完全損壞（no space left + 執行檔遺失）
VM 內部 filesystem 空間耗盡，/Docker/host/piped 報錯；MacOS/ 目錄下執行檔也遺失。→ 改用 Colima：brew install colima，2分鐘安裝完成，CLI 完全相容原本所有 docker 指令，且資源佔用更低。
Problem 02
LangFuse v3 需要 ClickHouse（複雜度過高）
拉最新 image 後 CLICKHOUSE_URL is not configured 報錯。→ 降版使用 langfuse:2：v2 只需 PostgreSQL，架構簡單，功能完整，適合本地自托管。
Problem 03
LangFuse v2 不支援 LANGFUSE_INIT_* 環境變數
想透過 docker-compose env 預置帳號 + API keys，但 v2 完全不支援這個功能。→ Playwright 自動化：自動打開 localhost:3010，填入帳號資料，建立 project，複製 API keys，寫入 Kai .env。
Problem 04
LangFuse Python SDK v4 API 完全不相容
SDK v4 改用 OpenTelemetry，__init__() 參數完全不同（enabled= 不存在）。→ 捨棄 SDK，改用 urllib.request 純 HTTP：直接打 /api/public/traces 和 /api/public/ingestion，零外部依賴，且永遠不會有 SDK 版本升級問題。

Section 01

執行摘要

Klay 策略分析與 Ray 架構審查的三項最核心共識：

01
方向正確，但定義要改：Auto Research 不是「建新系統」，而是「給現有 Planning → Execution → Review 管線加一層量化 feedback 迴路」。現有系統缺少 iteration loop 和量化 metrics，這才是要補的。
02
三場景不可同時做：Coding Self-Healing Loop 優先（metric 最清楚、基礎設施已就緒、零財務風險）；智慧爬蟲優化其次；Stockcast 策略優化暫緩（overfitting 風險 + 缺少 walk-forward validation infra）。
03
7 項護欄缺一不可：Ray 指出最高風險是「無監督成本失控」與「AI 自我評分偏差」。M1-M7 必要條件必須在啟動前全部落地，否則 Auto Research 比現有人工管線更危險。

Section 04

Ray 風險矩陣

Ray 架構審查識別的主要風險（標記 R1-R6）：

HIGH

R1 — 成本失控

無監督 loop 每次迭代消耗 tokens + compute，沒有 per-loop cost tracking 就無法在爆超上限前停止。解決方案：硬性上限 50K tokens + 30min timeout（M3）。

HIGH

R2 — 自我評分偏差

LLM judges LLM，AI 傾向給自己高分。evaluate() 必須用客觀 metric（test pass/fail count），禁止 LLM 自評（M2）。

HIGH

R6 — Stockcast Overfitting

10 次迭代足以 fit 歷史資料但毫無預測力。在 walk-forward validation infra 建立前，Stockcast 場景不應進入 Auto Research 框架。

MEDIUM

R3 — refine() 基於 latest 而非 best

如果 refine() 基於最新一次結果而非歷史最佳結果，錯誤方向的 iteration 會越改越偏。必須改為追蹤 best_result（M5）。

MEDIUM

R4 — 結果未持久化

Iteration 中途 crash → 所有嘗試記錄消失，等於白做。需要每次迭代 JSON log 持久化（M6）。

MEDIUM

R5 — 黑箱執行

Auto Research 跑完沒有整合到 Kai task board，Rain 無法追蹤進度和結果。需要整合 Kai（M7）。

LOW

R7 — 缺少 per-iteration timeout

單次 iteration 無時限，可能因意外阻塞整個 loop。需要加 timeout 保護（已含在 M3 中）。

⚔️ Ray Skeptic Check — 魔鬼代言人

「如果根本不做這個呢？」現有人工改善管線（lessons.md → session review → SKILL.md）已在運作，並非完全空白。Auto Research 是加速器，不是填補空洞的必需品。

「更簡單的替代方案？」直接 retry wrapper（test fail → re-dispatch John → re-run）就能解決 80% 的 Coding Self-Healing 需求，不一定需要泛化的 10 次迭代框架。

「團隊最大瓶頸是什麼？」不是「迭代速度慢」，而是「任務太多」。Auto Research 應該聚焦在真正有 failing metric 的場景，而不是為了「有自動化」而做。

Section 06

7 項必要條件（M1-M7）

Ray 簽核的前提條件，全部落地後才可開始 Phase A 實作：

M1

唯一 MVP 場景

禁止三場景同時做。Phase A = Coding Self-Healing Only。三場景同時進行 = 三個方向的 token 浪費，且互相干擾。

M2

客觀 evaluate() 函數

evaluate() 必須基於客觀 metric（test pass count / 爬蟲成功率），絕對禁止 LLM 自評。LLM 自評偏差已是業界共識，不可依賴。

M3

硬性成本上限

50K tokens per loop + 30min timeout。超過任一上限 → 強制停止 + Discord 通知 + Kai task 標記。無上限 = 無監督成本失控。

M4

Early Stopping 邏輯

連續 3 次無改善 → 自動停止。比 max_iterations 更重要：防止在已達瓶頸後繼續消耗資源。

M5

refine() 基於 best_result

refine() 必須基於歷史最佳結果，而非最新一次結果。基於 latest 會導致「退步後繼續錯誤方向優化」，最終遠離最佳解。

M6

結果持久化

每次迭代產出 JSON log（含 metric、timestamp、token 消耗、改動 diff）。Crash 後可 resume，也可事後 audit。

M7

整合 Kai Task Board

Auto Research 每次 loop 必須建立 Kai task + 回報進度 + mark done。不能是黑箱。Rain 必須能從中控台看到 Auto Research 的執行歷史。

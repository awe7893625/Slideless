03 — 問題一：Jarvis Telegram 指令是否自動登記任務看板？
部分自動  取決於 Jarvis 是否調用 dispatch_klay/ray/worker
✅
有登記：dispatch_klay / dispatch_worker / dispatch_ray
呼叫 spawn-klay.py / spawn-ray.py / spawn-worker.py， 腳本內部呼叫 Kai API 建立任務，並設 auto_start: true（直接進入 executing 狀態）。 任務完成時由 Klay/Ray 在 prompt 中呼叫 /done endpoint。
⚠️
無登記：dispatch_openclaw（外部 agents）
當 Jarvis 呼叫 dispatch_openclaw（spider/lynx/probe/echo/noel/buzz）， 目前沒有對應的 Kai 任務建立，Rain 無法在看板追蹤這些外部任務的狀態。 建議後續改善：openclaw dispatch 前後各一次 Kai 任務建立/完成。
ℹ️
設計上不登記：Jarvis 自己處理的訊息
簡單聊天、投影快捷（/mirror）、直接查詢等，Jarvis 自行回應，不需要登記為任務。 這是預期行為。
04 — 問題二：任務看板「進行中」欄位顯示異常
根本原因找到  session-start.sh 孤兒任務 + 非預期 claim+done 失敗
🔍
根本原因 1：Terminal Session 任務 claim+done 失敗孤兒化
~/.claude/hooks/session-start.sh 在每次 Claude Code session 啟動時建立 "Terminal Session" 任務，然後立即 claim + done。 但當 Kai 回應慢（--max-time 5）或連線不穩時，done 呼叫失敗， 任務卡在 pending，永不完成，積在「待處理」欄位。

影響：連續 12 個 Terminal Session 任務堆積在 pending， Rain 看到「待處理」欄位 12 筆，誤以為有真實任務未處理。
🔍
根本原因 2：Stale Executing 任務（spawn-ray 孤兒）
spawn-ray.py 建立 Kai 任務後啟動 tmux Ray session，若 Ray session 提早結束 （完成審查後未呼叫 /done 或 Klay 未等到結果），任務卡在 executing， 需等待 30 分鐘 timeout 才自動 fail。

影響：「執行中」欄位出現 progress=0% 的假執行任務，誤導 Rain 以為有任務卡住。
✅
前端邏輯正確（非問題所在）
TaskBoard.tsx 正確將 claimed 狀態映射到「執行中」欄位， 後端每 60 秒掃描一次 timeout，1800 秒後自動標記 FAILED。 前端無需修改。
05 — 已執行修復
FIX 1 清理 12 個 stale Terminal Session pending 任務 — 全部呼叫 claim + done 標記為 success（audit trail 保留）。
FIX 2 修復 session-start.sh：done 呼叫失敗時，改為呼叫 DELETE 刪除任務， 避免孤兒任務堆積在 pending 欄位。
FIX 3 強制將 stale Ray 審查 executing 任務（0bb5574a）標記為 failed， 清空「執行中」欄位。

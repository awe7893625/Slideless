Chapter 01

三種 Session 類型

Rain AI 團隊的所有工作都透過三種不同的 Session 模式進行。每種模式有不同的觸發方式、執行流程和適用場景。

Claude Code · Orchestrator

普通 Session

  • Rain 在終端機直接下指令
  • Orchestrator 輸出 Scope Card
  • 依 Grade 決定行動路徑
  • 用 Agent tool 呼叫子 agent
  • 子 agent 完成後回傳結果
Telegram · klayt tmux

Klay Telegram Session

  • Rain 在 Telegram 傳訊息
  • klayt 收到後先回「收到」
  • 建立 Kai task 再進行規劃
  • edit_message 模擬進度更新
  • 完成後發新訊息觸發通知
  • heartbeat 每 30s 更新狀態
Noel · Gemini 2.5 Pro · 凌晨 1:00

夜班 Session

  • night-shift-prep.py 建任務
  • 固定：健康檢查 / 知識管線
  • 固定：LaunchAgent 巡檢
  • 輪替：7天循環維護不同專案
  • dispatch Tier 3 外部 Agent
  • 晨間 Email 彙整報告

Chapter 02

Scope Card 與任務分級

每個任務開工前，Orchestrator 必須輸出 Scope Card。這是任務分級的唯一依據，基於客觀數字，不做主觀判斷。

2.1 Scope Card 格式

[SCOPE CARD]
Task:   一句話描述任務目標
Files:  逐一列出預估要改的檔案（不確定寫 unknown）
Layers: Frontend / Backend / DB / Config / Infra
Hours:  預估完成時間（如 2h / 4h）
Grade:  Tiny / Standard / Major / Critical

2.2 升級規則（客觀數字，自動判斷）

條件 自動升級到 說明
Files 列表 ≥ 4 個 Major 多檔案修改表示跨模組影響
Layers ≥ 2 種 Major 跨技術棧，需要前後端協調
Files 含 unknown + Hours > 2h Major 不確定 = 複雜，保守評級
包含 auth / payment / migration 關鍵字 Critical 高風險操作，需 Ray veto 保護

2.3 任務等級對應行為

Tiny

單檔修改、1-2 行 bugfix、config 調整

Klay 直接處理，無需 dispatch

Standard

一般 CRUD、單一 API endpoint、單一 component

Dispatch 單一 agent，Klay 不實作

Major

2+ 檔案 + 跨層修改、新模組

強制 3-Phase + 並行 dispatch

Critical

auth / payment / schema migration / infra

3-Phase + Ray veto 簽核

核心原則：Standard 以上，Klay 只做規劃和 dispatch，不自己 implement

Scope Card 的數字（Files 數量、Layers 數量）決定等級，Klay 不做主觀判斷。違反此規則會在 session review Q4 中被記錄，連續 3 次觸發 Klay audit。

Chapter 03

大型任務的 3-Phase 流程

Major 和 Critical 任務必須經過三個階段規劃，由三個不同的 agent 依序執行，確保商業、技術、風險三個維度都被充分審查。

3.1 三個階段總覽

Phase 1 — Klay
Claude Opus · 商業規劃
→
Phase 2 — Scout
Claude Sonnet · 系統架構
→
Phase 3 — Ray
Codex CLI · 風險審查
↓
並行 Dispatch 實作
（Max 後端 + John 前端，各自 git worktree）

3.2 各 Phase 詳細說明

Phase 1 · Claude Opus · Klay（CEO）

商業規劃層

確立任務的商業目標、範圍邊界和成功指標。Klay 在這個階段做決策，之後不再干預實作。

輸出：① 專案目標（一句話）② 約束條件（技術/時間/預算）③ 成功指標 ④ 範圍（做什麼/不做什麼）⑤ 優先級（P0-P3）⑥ 安全與架構預檢（auth 設計、模組策略、環境變數清單）
Phase 2 · Claude Sonnet · Scout（系統架構師）

系統架構層

基於 Phase 1 的商業規格，設計技術實作方案。Scout 只做設計，不自行調整需求範圍。

輸出：① System Components 表 ② API Contracts（端點/請求/回應）③ Data Models（DB schema）④ Integration Points（修改點）⑤ Dependencies（外部服務/環境變數）⑥ Implementation Order ⑦ Recommended Skills

網站任務同步觸發：Webb（規格包）+ Iris（UI/UX 規格）+ Silo（SEO 結構）

Phase 3 · Codex CLI · Ray（設計審查官）

風險審查層

Ray 審查設計的正確性與風險，不審程式碼品質。Critical 任務有一票否決權（veto）。

輸出：① Task Definition Check ② Risk Assessment（清單+嚴重度）③ Edge Cases ④ Skeptic Check（「不做這個呢？」「有更簡單替代？」）⑤ Architecture Sign-off（✅/❌/⚠️）⑥ Requirement Traceability（Major/Critical 強制）

3.3 並行實作：worktree 隔離

⚙ 雙人並行 dispatch（符合以下任一條件自動升級）
觸發條件 主力 配對 分工
複雜後端（5+ 檔案） Max（架構/邏輯） Rex（CRUD/API） Max 做核心，Rex 做周邊 endpoint
前端重度（3+ 前端檔） John（邏輯/狀態） Iris（CSS/RWD） John 做功能，Iris 做 UI 層
全端功能（前後端同時） Max（後端） John（前端） 各自在獨立 worktree，最後 merge

⚠️ 兩個 subagent 必須修改不同檔案才能並行，否則退回單人順序執行

3.4 衝突裁決規則

衝突類型 最終決定權 例子
產品方向 / 商業決策 Klay 做不做某功能、MVP 範圍、優先順序
技術實作方案 Scout API 怎麼切、service boundary、DB schema
安全 / 穩定 / 回滾風險 Ray（有 veto 權） auth、payment、migration、production incident risk
Chapter 04

3 層審查流程

所有 Standard 以上的任務，在實作完成後必須通過三層審查才能部署。三層各司其職，不重疊。

L1

Quill — 程式碼審查（Code Review）

審查 PR/diff、規格合規性、可讀性、DRY 原則、測試缺口。Standard 任務做 Lightweight review，Major/Critical 做 Full review。
Quill 不做：執行 lint/test、架構決策、veto。

Standard → Lightweight Major → Full Critical → Full
L2

Forge — CI Gate（持續整合驗證）

執行 lint / typecheck / unit test / build / preview deploy。所有部署前必過。Major 以上需額外提供 Known Issues 清單和依賴變動清單；Critical 需 Rollback plan + Smoke test result。
Forge 不做：程式碼品質審查、架構決策。

所有部署前強制
L3

Ray — 設計審查（Pre-Implementation，Phase 3）

審查任務定義正確性、方案架構風險、邊界情況。Critical 任務有一票否決權。Ray 在實作前（Phase 3）執行審查，不在事後審程式碼。
Ray 不做：lint/typecheck、PR diff review、命名可讀性。

Major → Full 審查 Critical → Full + Veto 權
自動

Sentinel — git hook 密鑰掃描

所有 commit 自動執行，掃描 20 種 secret pattern + 15 種敏感檔名。警報後不可跳過，即使 Quill/Forge/Ray 已通過。

所有 commit 自動
手動

Probe + Echo — UI 驗收（Chrome MCP）

桌機（1280×900）+ 手機（390×844）UI 驗收。Forge CI 偵測到 payment/auth/RWD/3+ 前端檔案變更時標記 needs-ui-test，由 Klay 決定是否觸發。每日上限 3 次。

Klay 判斷後手動觸發
顧問

Codex co-review — Advisory Pass（2026-03-21 新增）

Major/Critical 任務在 Quill Full review 完成後，自動追加 Codex CLI advisory pass。結果僅供參考，不阻擋 pipeline。Tiny/Standard 豁免。

Major/Critical 自動 結果不阻擋 pipeline

4.1 審查矩陣（任務等級 × 審查層）

任務等級 Sentinel Quill (L1) Forge (L2) Ray (L3) Probe+Echo Codex Advisory
Tiny ✅ 自動 ❌ ❌ ❌ ❌ ❌
Standard ✅ 自動 ✅ Lightweight ✅ ❌ ❌ ❌
Major ✅ 自動 ✅ Full ✅ ✅ Full 手動觸發 ✅ Advisory
Critical ✅ 自動 ✅ Full ✅ ✅ Full+Veto 手動觸發 ✅ Advisory
Chapter 05

Codex CLI 在團隊的應用

Codex CLI 在 Rain AI 團隊中承擔「審查端」的核心角色。設計的核心思想是 maker-checker 分離——不同模型互相制衡，避免同一個模型既設計又審查的盲點。

Maker 端（設計與實作）

  • Klay（Claude Opus）—— Phase 1 商業規劃
  • Scout（Claude Sonnet）—— Phase 2 架構設計
  • Max（Claude Opus）—— 後端實作
  • John（Claude Sonnet）—— 前端實作
  • Rex / Iris —— API / UI 實作

Checker 端（審查與制衡）

  • Ray（Codex CLI） —— Phase 3 設計審查
  • Ray（Codex CLI） —— veto 否決權
  • Codex Advisory Pass —— 事後程式碼 advisory
  • Quill —— 程式碼品質（Claude Sonnet）
  • Sentinel —— 密鑰掃描（git hook）

5.1 Ray：Codex CLI 的核心應用

R Ray — 方案架構審查官（Codex Plus CLI）

Ray 整個 agent 跑在 Codex Plus CLI 上。所有 Major/Critical 任務的 Phase 3 設計審查、架構風險評估、開工簽核都是 Codex 執行。

Ray 做什麼

  • 任務定義正確性（需求有沒有漏？目標是否明確？）
  • 方案架構風險（設計是否合理？邊界情況？）
  • 高風險設計偵測（是否應升為 Critical？）
  • Skeptic Check 魔鬼代言人（主動挑戰假設）
  • 開工簽核（✅ 通過 / ❌ 打回 / ⚠️ 有條件通過）
  • auth/payment/migration 的一票否決權（veto）

Ray 不做什麼

  • 程式碼品質（命名、可讀性 → Quill）
  • lint / typecheck / pytest 執行（→ Forge）
  • 密鑰掃描（→ Sentinel hook）
  • 日常 post-implementation code review（→ Quill）

5.2 Codex co-review：Advisory Pass（2026-03-21 新增）

A Codex Advisory — 事後程式碼顧問審查
Quill Full Review 完成
→
自動觸發 Codex Advisory Pass
→
結果標記為「僅供參考」
↓
不阻擋 pipeline
→
設計衝突 → Ray 決定
/
程式碼衝突 → Quill 決定

Tiny/Standard 任務豁免，不觸發 Advisory Pass。

5.3 為什麼選 Codex 作為審查端？

Maker-Checker 分離原則

Claude（Klay/Scout 設計，Sonnet/Opus 實作）和 Codex（Ray 審查）是不同的模型，各有不同的訓練偏差和盲點。讓設計者和審查者使用不同模型，可以有效避免「自我確認偏誤」—— 即同一個模型對自己的設計過於寬容的問題。

衝突時最終決定權：設計問題 → Ray（Codex），程式碼問題 → Quill（Claude Sonnet）

5.4 Codex 應用全覽

應用場景 角色 觸發條件 結果是否阻擋
Ray Phase 3 設計審查 Ray（Codex Plus） Major/Critical 任務開工前（Phase 3） ✅ 會（可 veto）
Codex co-review Advisory Codex CLI（anonymous） Quill Full Review 完成後（事後） ❌ 不會（僅參考）
Chapter 06

Hook 自動化系統

Hook 是 Claude Code 的事件驅動自動化機制，掛載在 ~/.claude/settings.json。共四個事件層，涵蓋 Session 全生命週期的自動化。

6.1 SessionStart — Session 開始時

Hook 作用
session-start.sh 讀取上次 handoff 檔案，載入前次工作 context
session-start-handoff.sh 檢查是否有未完成任務需要接續

6.2 PreToolUse — 每次工具呼叫前

PreToolUse hook 在工具執行前攔截，可回傳 non-zero exit code 阻止執行。

Hook 觸發工具 作用
scope-card-gate.sh Read / Bash / Edit / Write / Glob / Grep 開工前確認有輸出 Scope Card，否則攔截
dispatch-gate.sh Edit / Write 確認 Standard+ 任務有 dispatch，不可 Klay 自己做
block-random-md.sh Write 禁止隨意建立 .md 文件（防止亂建文件）
config-protection.sh Edit / Write 保護設定檔不被誤改
deck-quality-gate.sh Write 阻擋低品質 Deck HTML（必須由 Nova 製作）
scan-secrets.sh Bash 每次執行 Bash 前掃描密鑰洩漏風險
block-dangerous.sh Bash 攔截危險指令（rm -rf / force push 等）
warn-push.sh Bash git push 前警告確認
heartbeat-auto.sh 全部工具 記錄 agent 活動心跳

6.3 PostToolUse — 每次工具呼叫後

PostToolUse hook 在工具執行後觸發。async hook 背景執行，不影響主流程。

Hook 觸發工具 模式 作用
activity-logger.sh 全部 async 記錄所有工具使用到 activity log
auto-format-v2.sh Edit / Write async 自動格式化修改過的程式碼
warn-console-log.sh Edit / Write async 偵測 console.log 未移除，發出警告
compact-reminder.sh Edit / Write async context 接近上限時提醒 compact
dispatch-reminder.sh Edit / Write async 修改 3+ 檔案時提醒 dispatch（3 檔 ⚠️ 5 檔 🔴）
typecheck-debounce.sh Edit async 編輯後自動跑 TypeScript typecheck（防抖）
log-pr-url.sh Bash async 記錄 PR URL 到日誌
error-capture.sh Bash async 捕捉錯誤寫入 lessons.md（自我改進）

6.4 Stop — Session 結束時

Hook 模式 作用
require-session-review.sh 同步（阻擋） 強制五問 Session Review，不完整就無法結束
verify-before-done.sh async 確認宣稱「完成」前已執行驗證
notify-done.sh async 發送完成通知
save-session-log.sh async 儲存本次 session 完整記錄
auto-handoff.sh async 自動寫入 handoff 檔，供下次 session 接續
session-activity-tracker.sh async 統計本次 session 活動數據

6.5 關鍵設計原則

同步 vs 非同步 Hook

同步 Hook（無 async 標記）→ 可攔截行為。回傳 non-zero exit code 即阻止後續執行，是真正的「護欄」機制。

非同步 Hook（async: true）→ 背景執行，不影響主流程。用於記錄、通知、格式化等不需阻擋的副作用。

require-session-review.sh 是唯一掛在 Stop 的同步阻擋 Hook，確保每個 session 都留下五問回顧記錄，形成強制的品質閉環。

S

SessionStart（2 hooks）

載入前次 context，檢查未完成任務。確保 session 有完整的工作記憶接續。

P

PreToolUse（9 hooks）— 安全護欄

Scope Card 強制、dispatch 合規、危險指令攔截、密鑰掃描、文件建立管控。任何 hook 回傳 non-zero → 工具呼叫被取消。

P

PostToolUse（8 hooks）— 品質自動化

格式化、型別檢查、dispatch 提醒、錯誤捕捉、PR URL 記錄。全部非同步執行，不影響主流程速度。

E

Stop（6 hooks）— Session 閉環

Session Review 強制（同步阻擋）、驗證確認、通知發送、日誌儲存、handoff 寫入、活動統計。確保每個 session 有完整生命週期記錄。

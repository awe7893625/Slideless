為什麼晨報一直寄信箱？
根本原因 + 修復方法
原因 1：我們測試時手動跑了腳本 3 次
開發過程中 bash morning-report.sh 執行了 3 次，每次都送了一封 email。LaunchAgent 本身只在 09:00 跑一次，正式上線後一天只有一封。
原因 2：Discord Webhook URL 未設定 → 永遠走 Email fallback
腳本邏輯：先找 Discord → 找不到 → 寄 Email。Kai .env 目前沒有 DISCORD_WEBHOOK 這個 key，所以每次都落到 Email。
修復方法（二選一）
方法 A：設定 Discord Webhook（推薦）
1. Discord → 目標頻道 → Edit Channel → Integrations → Webhooks → New Webhook → Copy URL
2. 把 URL 加入 Kai .env：DISCORD_WEBHOOK=https://discord.com/api/webhooks/xxx/yyy
3. 重啟 Kai：bash restart_simple.sh

方法 B：接受 Email 模式（什麼都不改）
每天 09:00 只收一封 Email 晨報。測試時若不想收信，不要手動執行腳本即可。

步驟級改善回路（新建立）
進步不只在任務結束時，每個執行步驟都有改善空間
設計理念
舊機制只在任務完成後觸發改善（三問、lessons.md）。但每個執行步驟本身——用哪個 tool、哪個 skill、哪個順序——都有優化空間。這些微層級的改善需要一個即時記錄管道。
改善回路設計
執行中任何步驟 發現更快/更好方式
→
立即記錄 speed-improvements.md
（不等任務結束）
→
隔日晨報曝光 ⚡ 效率亮點區塊
（全團隊可見）
→
Noel 週報驗證 驗證 3+ 次的項目
→
升級到 SKILL.md 成為團隊標準操作
類型舊做法新做法效益
[TOOL] Bash find / ls Glob tool 按修改時間排序，免 shell 啟動
[TOOL] Bash grep/rg Grep tool 有 -A/-B/-C context，output_mode 控制
[TOOL] Read + Write 整檔 Edit（diff only） 節省 80-90% token，降低覆蓋風險
[FLOW] 序列 tool calls 並行 tool calls（同一 response） N 個獨立操作 → 幾乎同時完成
[SKILL] 序列 Agent 派遣 parallel agents + run_in_background 3 個 Agent 並行 = 1/3 total time
[AGENT] Explore（廣搜） scout subagent（精確架構分析） 結果乾淨，直接輸出 spec 格式
[API] curl URL 不加引號 curl "URL?param=val" 永遠加引號 避免 zsh glob 展開 bug
[FLOW] bash → python3 複雜 escape KEY="val" python3 傳 env var 支援任意 unicode，消除 escape hell

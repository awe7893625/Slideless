SLIDE 02 · OVERVIEW

什麼是 Board？

Board（董事會）是 AI 團隊的策略諮詢層，專門解決「要做什麼」的問題。

在 AI 團隊中，現有的 3-Phase 和 Debate Protocol 都是解決「怎麼做」的工具。Board 填補了一個更上游的空缺：在開始設計之前，先確認我們要走哪條路。

🏛️ Board 策略層 → 決定 WHAT / 方向選擇 / 多方案評估
↓ Rain 裁決後
⚔️ Debate Protocol 執行層 → 決定 HOW / 架構辯論 / Challenger Step
↓
📐 3-Phase 工程層 → Klay + Scout + Ray 設計 + 正式風險簽核
↓
🔨 實作 Max / Rex / John / Iris 並行執行
⚠️ Board ≠ 取代 3-Phase。兩者串行，不競爭。Board 在上游，3-Phase 在下游。
核心價值：在錯誤的方向上全速前進，是最昂貴的浪費。Board 確保你在「砍掉重練」之前先問「這值得做嗎？」以及「有更好的路嗎？」

SLIDE 04 · TRIGGERS

觸發條件

兩種觸發路徑 + 明確的不觸發排除清單，避免不必要的開會。
🤖

自動觸發（兩個條件缺一不可）

① 任務標記為 Critical
② 存在明確的策略方向選擇：多方案 / 定價 / 架構重構 / 新產品方向

👤

手動觸發（任一指令）

/board [topic] 或 "召開董事會 [topic]" 或 "幫我評估這個想法 [idea]"

⏭️

不觸發（直接走 3-Phase）

純技術 Critical（security hotfix、migration、bug fix）→ 方向已確定，只需執行

✅ 適合召開 Board

  • 商業模式選擇（B2C vs B2B）
  • 技術棧轉移決策
  • 定價策略（新產品上線）
  • 新部門 / 新功能是否值得做
  • 架構重構 vs 維持現狀

❌ 不適合召開 Board

  • Security hotfix（緊急，時間敏感）
  • DB migration（技術確定，方向清楚）
  • Bug fix（沒有策略選擇）
  • 小功能迭代（Standard 任務）
  • 已有明確 Rain 指示的任務
SLIDE 05 · WORKFLOW

四步工作流

總耗時 ~35-45 分鐘，Rain 收到報告後自行裁決，不阻擋夜班執行。

執行者：Klay · ~10min
背景簡報（500 字以內），說明觸發原因、現有資訊、未知項目
Step 02

執行者：Klay / Opus · ~15min
2-3 個方案，每個含成本 / 風險 / 擴展性 / 主要風險 1-2 點
Step 03
並行稽核
執行者：Vick + Ray · ~10min
《董事會提案與風險報告》— 工程初評 + 異見挑戰
Step 04
Rain 裁決
執行者：Rain · 隨時
選擇方案 → Klay dispatch 實作，進入正式 3-Phase 流程

SLIDE 07 · NIGHT SHIFT

夜班處理規則

Noel 夜班遇到 Board 觸發條件時，完成準備但不阻擋執行。

完成 Step 1-3，準備《董事會提案報告》

Noel 主持脈絡蒐集 + 提案 + 並行稽核（Vick + Ray），整合成完整報告

標記「等待 Rain 晨間裁決」，不阻擋實作

報告準備好但 Rain 未裁決 → 任務狀態標記 PENDING_DECISION，夜班繼續跑其他任務

Discord + Email 通知 Rain

自動推送 Discord DM + Email 給 Rain，附上報告連結和建議期限

夜班繼續執行其他任務

Board 觸發的任務暫停，不因等待裁決而讓夜班空轉

Rain 早上裁決後，Klay 接手 dispatch

Rain 回覆選擇方案 → Klay 立即建立任務 → 進入正式 3-Phase 流程

關鍵設計原則：Board 不應成為夜班的阻塞點。準備好報告、通知 Rain、繼續跑其他任務 — 這是正確的節奏。
SLIDE 08 · DESIGN DECISIONS

設計決策記錄

Ray Phase 3 審查通過（Conditional Pass），解決了兩個 Blocker。
BLOCKER 01

Gemini Pro 調用

問題
Claude Code Agent tool 不原生支援 Gemini，直接調用會失敗
解法
Klay 親自做 Step 1 脈絡蒐集；Gemini Pro 降為 optional，只有 Rain 明確要求才啟用
BLOCKER 02

觸發衝突

問題
Critical 任務已自動觸發 3-Phase，Board 也自動觸發 → 雙重觸發衝突
解法
Board 自動觸發需要兩個條件同時成立（Critical + 策略方向選擇）；純技術 Critical 直接走 3-Phase
決策 結果 理由
Gemini Pro 的角色 降為 Optional Claude Code 不原生支援跨廠商 Agent；技術限制優先於理想設計
Board vs 3-Phase 觸發 雙條件門檻 Critical 不等於需要 Board；加入「策略方向選擇」作為第二條件
Ray 在 Board 的角色 Advisory Only Board 的 Ray 是初步評估，不取代後續 3-Phase 的正式 Ray Phase 3 簽核
夜班阻擋機制 非阻擋式 準備報告後繼續跑，等待 Rain 晨間裁決，不讓夜班空轉

## 🏛️ 董事會提案報告 — StockSignal 商業模式選擇 召開日期: 2026-03-24 觸發方式: Rain 手動（/board） ### 方案 A：B2C 個人訂閱（NT$299/月） - 實作成本: Standard - 架構影響: 需要 Stripe/ECPay 訂閱機制、用量計費 - 擴展性: 高（SaaS 規模化，邊際成本低） - 主要風險: 台灣市場散戶付費意願低，取得成本高 ### 方案 B：B2B 券商白標（年費 $50K） - 實作成本: Major - 架構影響: 需要 API 安全設計、自定義品牌、客戶隔離 - 擴展性: 中（客製化需求高，難以標準化） - 主要風險: 銷售週期長（3-6 個月），需要業務團隊 ### 🔍 稽核報告 ⚔️ Vick（異見）: 方案 A 低估了取得成本，台灣付費轉換率 < 5%， 月活躍用戶需要 5,000+ 才能打平。方案 B 的年費模式現金流更穩定。 🔧 Ray（工程初步評估）: 方案 B 需要 API 安全性設計， Phase 3 必須檢查 API Key 輪替、客戶資料隔離、SLA 保證條款。 ### 💡 Klay 建議 方案 A 先驗證市場（3 個月 MVP），若 DAU < 100 → 轉向方案 B Rain 裁決: ⏳ 等待決策
下一步：Rain 回覆「選方案 A」或「選方案 B」→ Klay 立即建立 Kai task → 進入正式 3-Phase（Scout 架構 + Ray 簽核）→ 實作。

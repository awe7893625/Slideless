根因解剖 MECE
根因一 — 兩份分歧副本
線上管線執行的是 Stockcast repo 副本，而非 ~/Projects/scripts 副本
generate_stockcast.py 透過 Path(__file__).parent 解析 podcast-cache-builder.py——即 Stockcast repo 自己的子目錄。 先前的修復只更新了 ~/Projects/scripts/podcast-cache-builder.py，Stockcast repo 內的線上副本從未被觸及。 每次後續執行仍使用損壞的解析器，且毫無警示。

解析器漏掉無括號公司名、拒絕美股代碼、缺少情緒詞彙

修復清單 全部現場確認
修復項目變更內容為何能防止復發
容錯區塊感知解析器 以容忍無括號公司名、美股代碼及擴充情緒詞彙（「看好」、「看空」等）的區塊解析器取代原有的逐行解析器 能處理集數格式的結構性差異，不會因格式變化而丟棄股票

位元組一致副本同步 ~/Projects/scripts/podcast-cache-builder.py 與 Stockcast repo 副本現在在位元組層級完全相同 永久消除「修錯副本」這整類分歧缺陷

缺陷發現：.TW vs .TWO 符號錯誤
問題根因
8299.TW 在 Yahoo Finance 回傳 404；管線崩潰且無優雅錯誤處理
群聯電子（Phison）8299 掛牌於台灣證券交易所 OTC 市場（上櫃），其正確的 Yahoo Finance 符號為 8299.TWO，而非一般上市股票使用的 8299.TW。 管線直接以 .TW 後綴查詢，收到 404 後崩潰並拋出未捕獲的例外，無任何友善錯誤訊息。 此缺陷在實際分析台灣 OTC 掛牌股票時才得以浮現——純台灣上市股（.TW）不會觸發此問題。
修復方案：_yf_fetch 自動 Fallback 序列
Commit 57d46cd360
_yf_fetch：.TW ↔ .TWO + query1 ↔ query2 自動 fallback + 優雅錯誤處理
修復引入 _yf_fetch 輔助函式，實作下列四步驟 fallback 序列：

1. 嘗試 {ticker}.TW via query1.finance.yahoo.com
2. 若 404 → 自動切換至 {ticker}.TWO via query1.finance.yahoo.com
3. 若仍失敗 → 再試 query2.finance.yahoo.com 端點
4. 全部失敗時回傳結構化錯誤訊息，不崩潰管線——後續子系統可繼續執行

此修復永久消除了對上市 vs 上櫃符號差異的手動辨識需求。

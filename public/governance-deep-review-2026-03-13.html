Scout — 已修復 Risks（4 項）

9 個架構風險中，高/中優先的 4 項已即時處理。

R7 — L3 _fix_scheduler_unregistered_job 缺 job_key 驗證 High 已修復
未驗證的 job_key 可寫入任意排程設定，存在注入風險。
修復：加入 re.match(r'^[\w.-]+$', job_key) guard；驗證失敗時 fallback 並記錄 warning，不執行 fix。
R2 — 審查快取非原子寫入，kill 可能損壞快取檔 Medium 已修復
直接 open(path, 'w') 寫入，若程序在中途被中止，快取檔案變成半殘廢 JSON。
修復：改為先寫 .tmp 暫存檔，完成後 os.rename() 原子替換。
R5 — Module H timeout 60s 可能不夠 Low 已修復
健康檢查在高負載時可能超時，導致夜班報告缺少 H 模組資料。
修復：timeout 提升為 120s。
R6 — Module I 只顯示最後 5 筆 autofix 記錄 Low 已修復
密集修復期一夜可超過 5 筆，報告截斷。
修復：提升為 last 20 筆。
Slide 4

Scout — Architecture Gaps（v2 Roadmap）

12 個架構缺口本次均未修復，按優先順序進入 v2 Roadmap。

高優先（建議 v2 優先處理）

G1 — verified=True 用來代表審查完成不可靠 High Roadmap
H4 以 verified 欄位判斷任務是否通過審查，但 verified 也可由其他流程設定，語義不精確。建議：新增 review_evidence 欄位，記錄審查者 + 時間戳。
G6 — 無 API 端點觸發 autofix on-demand High Roadmap
目前 autofix 只能排程觸發，無法在問題發現後立即修復。建議：加入 POST /api/governance/autofix 端點，支援選擇性執行特定修復層。
G9 — 無 Sentinel Hook 運作審查 High Roadmap
Governance 系統未涵蓋 Sentinel git hook 的健康狀態，Hook 失效可能被忽視。建議：新增 H9 模組，每日驗證 Sentinel hook 是否掛載且可執行。
G10 — 無 3-Phase Planning 合規審查 High Roadmap
Major/Critical 任務要求完整三階段規劃，但 Governance 目前無驗證機制。建議：加入 Phase 規劃合規度到 governance roadmap。

中優先

G2 — H4 跳過 Standard 任務的 Quill/Forge 審查 Medium Roadmap
只審 Major+ 的 verified 欄位，Standard 任務是否完整通過 Quill + Forge 無從確認。
G3 — 高 unknown_severity 可膨脹合規率 Medium Roadmap
unknown_severity 任務被計入分母，拉高合規百分比，但這些任務實際上未受嚴格管控。建議分開報告。
G8 — 審查合規僅夜間偵測，無即時 hook Medium Roadmap
問題最早在第二天早晨才被偵測，高優先違規有 8–16 小時延遲窗口。

低優先

G4 — 無反向掃描

無從確認 agent 目錄是否與 team.json 完全對齊，ghost agent 目錄未偵測。

G5 — Module H 超時後無 cache 存在檢查

超時後報告顯示空白，而非回落到上次快取資料。

G7 — 無 per-module 獨立 API 端點

無法只執行特定模組（如只跑 H5），除錯與維護不便。

G11 — 無 frozen agent 活動偵測

Milo/Sage 已凍結，但若被誤用也不會觸發告警。

G12 — 無模型成本層級升級偵測

Sonnet 任務若被路由到 Opus 不會觸發告警，成本無法控管。

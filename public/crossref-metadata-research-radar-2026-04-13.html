Executive Take

這不是在教團隊怎麼繞過 Cloudflare，而是在改問題定義

真正有價值的點不是「前台抓成功」，而是把問題從脆弱、易壞、法律與維護成本都高的 UI scraping，轉成更上游、更結構化的 scholarly metadata ingestion。

What This Solves
  • 更早取得 DOI、標題、作者、期刊、deposit 時間等 metadata
  • 避開受 Cloudflare 保護的前台頁面與 challenge 流程
  • 用 DOI 當主鍵，後續串 PubMed、publisher page、知識庫、alert
  • 把 daily monitoring 做成增量同步，而不是天天全站重爬
What This Does Not Solve
  • 不能保證拿到全文
  • 不能取代所有 publisher-specific 欄位
  • 不能覆蓋非 scholarly content
  • 不能被誤包裝成「Cloudflare 已經被破解」
Why

官方證據其實都指向同一件事：不要跟展示層硬碰

Cloudflare、PubMed、Crossref 三方官方文件放在一起看，結論很清楚：前台 challenge 不穩，PubMed 有 deposit lag，而 Crossref 正好提供比較接近原始 scholarly metadata supply chain 的入口。

Source Official signal Engineering meaning
Cloudflare Challenge Pages Challenge 會檢查 browser environment 與 automated signals，甚至可能要求互動；而且 full HTML challenge 會破壞非 HTML/XHR 流程。 如果只是要 metadata，前台 scraping 通常是錯層打擊。
PubMed Help 文章發表到 PubMed 可查之間會受 publisher deposit 時間影響；PubMed 官方甚至說，固定巡檢新文時，`Create Date [crdt]` 常比 `Publication Date [dp]` 更完整。 PubMed 比較適合 enrich，不一定適合做最早警報源。
Crossref REST API 官方開放 REST API，不用註冊即可使用；works filters 支援 `from-created-date`、`from-update-date` 等，而且可用在 `journals/works` 這類 endpoint。 可以穩定做 journal-level incremental sync。
Use Cases

對 Rain AI Team 最有價值的 4 個落點

這條線不是只有文獻監控。真正值錢的是把 metadata 變成團隊的 daily research radar、知識入口與垂直情報產品底層。

A

Journal Monitoring

指定 journal、ISSN、主題詞的每日新文偵測。最直觀，也最接近這次 Blood 例子。

B

Research Radar

AI / agent / model / browser automation / observability 新研究雷達，接摘要、打標、推播與知識庫。

C

Vertical Intelligence

醫學、生技、材料、半導體等垂直情報產品的 daily update layer。

D

RAG Intake

先做 metadata ingest，再決定哪些 DOI 值得 deeper fetch / full text / vectorization，比盲爬省成本也更可控。

Not The Goal

不是全文爬取替代品

如果目標是全文、supplementary files、圖表、publisher-only rich content，Crossref 本身不是終點。

Team Fit

這不是從零開始的基建。從 repo 能力來看，watchlist、digest、Telegram、前台展示、knowledge writeback 都已經存在。

Architecture

最合理的系統長這樣：metadata first，enrichment second

核心設計原則是 incremental sync + DOI dedup。Crossref 先求穩定 intake，PubMed 與 publisher page 留在後段補強。

1. Scheduler + watchlists

以 journal ISSN、keyword、author、institution 為 watchlist 單位排程，固定跑 Crossref query。

2. Incremental sync

保存上次同步時間，使用 `from-created-date` 抓新 records，使用 `from-update-date` 抓補件與 metadata 修正。

3. DOI-first dedup

DOI 作為主鍵；若沒有 DOI，再退回 title + first author + container + year 的弱 dedup。

4. Enrichment + delivery

對值得關注的 DOI 再補 PubMed / publisher landing page / abstract parsing，最後推送 Telegram / Discord / Email，並寫入 Obsidian / SQLite / 前台展示。

curl -s 'https://api.crossref.org/journals/0006-4971/works?rows=3&sort=created&order=desc&select=DOI,title,container-title,created&mailto=rain@example.com'

不碰 `bloodjournal.org` 前台，只查 Crossref journal endpoint，就能拿到近期建立的 DOI / title / created timestamp。

  • sample DOI: `10.1182/blood.2025032810`
  • sample title: `GVHD fails the acid test`
  • sample created: `2026-04-09T15:37:15Z`
Meaning

對 daily monitoring 來說，這已經足夠驅動 alert、摘要、知識卡與後續 enrich。真正需要全文的項目，再往後段取，不必把最前面的 intake 綁在脆弱前台。

Decision

建議採納姿態：approve selective adoption

這條線值得做，但要嚴格定義用途，避免團隊把它誤講成萬用全文 ingestion。你們現在最該做的是把它掛到 research radar，而不是再去打一輪 journal frontend scraper。

Do
  • 先在 `ai-edu-media` 做 Crossref watchlist intake
  • 接 `投資Agent` 的 Telegram / research alert
  • 把 DOI 當主鍵，做後續 enrich
  • 把高價值結果寫回 Obsidian 與 RainAlpha research surface
Do Not
  • 不要先做 Cloudflare 對抗型 scraper
  • 不要把 Crossref 說成全文或 publisher rendering 替代品
  • 不要一開始就把 ingest、frontend、RAG、全文抓取一起做
  • 不要把這條線混成交易訊號本體

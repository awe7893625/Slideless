Chapter 02

Ray 提出了什麼建議

Ray 作為方案架構審查官，在本次審查中發現了 Klay 未察覺的關鍵 bug，並否決了兩個投資報酬率不足的提案。

P0 發現 — Klay 未察覺的 Bug

Ray 審查時發現 scan-secrets.sh 第 9 行使用 TOOL_INPUT="${TOOL_INPUT:-}" 讀取環境變數，但 Claude Code hooks 的輸入實際上是透過 stdin 傳入的，不是環境變數。

這意味著這個安全 hook 自建立以來從未正確執行過，Sentinel 密鑰掃描一直是失效狀態。

拒絕 Proposal A — 合併 Hooks

Ray 說明三個理由：

  • 目前 7 個 hook 使用三種不同的 blocking 機制（exit code 2、JSON decision:block、deny_with_reason），合併會造成語義衝突
  • 失去 failure isolation — 一個 hook crash 不應影響其他 6 個
  • 真正的 bottleneck 是 python3 啟動時間（30ms），而非 subprocess 數量，換 jq 即可解決
拒絕 Proposal C — 精簡 session-start

Ray 指出兩點核心理由：

  • 訂閱制下 Claude Code token 是零成本（feedback_subscription-cost-model.md 記錄）
  • BLOCKER 文字存在的原因是 Klay 曾在沒有它時跳過任務登記被糾正（lessons.md 記錄）

移除高回歸風險，節省成本為零，不值得做。

條件通過 — Module K 分頻

Ray 同意分頻方向，但反對「純週一完整，其他日完全不跑」，建議改為：

輕量每日（WoW 異常偵測）+ 完整週一（含 query 分析）

避免 SEO 異常偵測延遲超過一天。

Skeptic Check — Ray 的魔鬼代言人
「如果根本不做這些優化呢？Hook 系統一直都在工作，沒有 session 因為 hook 延遲失敗。scan-secrets 雖然壞了，但我們也沒有發生 secrets 洩漏事件。Token 消耗在訂閱制下是零成本。」 — Ray，Skeptic Check 必要輸出
Ray 的結論

P0 bug 無論如何必須修（這是安全護欄壞了），P1 jq 替換是低風險高品質的改善，P2 是合理的調整。其他提案投資報酬率不足，不做。

Chapter 05

未執行的項目與原因

兩個被 Ray 否決的提案記錄在此，作為未來決策參考。

Hook 合併（Proposal A）— 不做

Ray 說找到了更好的替代方案（python3 → jq），合併的風險大於收益。

如果未來 hooks 繼續增加到 15+ 個才需要重新考慮合併策略。目前 7 個 hooks 的規模不構成架構問題。

session-start.sh 精簡（Proposal C）— 不做

訂閱制下 token 是零成本，BLOCKER 文字是行為規範護欄，不應為了節省 0 成本而冒回歸風險。

這個護欄的存在有明確的歷史原因，記錄於 lessons.md，不應在沒有替代方案的情況下移除。

決策原則小結

本次優化的選擇框架：

  • 有量化收益 + 低回歸風險 → 執行
  • 零收益 + 有回歸風險 → 不做
  • P0 安全問題 → 無條件執行，不考慮 ROI
  • 架構變更 → 需要 Ray 顯式批准，不因「看起來更整潔」而做

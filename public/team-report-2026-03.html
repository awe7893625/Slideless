05 — Review Pipeline
審查管線：Quill + Ray
程式碼審查（Quill）與方案設計審查（Ray）完全分離 — 不重疊、不互相取代。
📝
Quill — Code Reviewer
PR/diff 審查 · 規格合規 · 可讀性 · DRY · 測試缺口
Claude Sonnet 4.6

審查等級

  • Lightweight（Standard 任務）：只查規格偏離 / 明顯 bug / blocking issue
  • Full（Major/Critical）：規格合規 + 可讀性 + DRY + 測試缺口 + 效能（O(n²)/N+1）
  • Quill 自行判斷走哪級，不需問 Klay

輸出格式

  • 判決：Pass / Needs Work / Rework
  • 問題分級：Blocker / Warning / Suggestion + 必須附 file:line 引用
  • 規則：不可有模糊回饋 — 必須引用具體行號

Quill 不做什麼

  • 商業決策 → Klay
  • 最終架構簽核 / veto → Ray
  • CI 執行 → Forge
  • 安全審計 → Sentinel
  • UI 驗收 → Probe+Echo
🔍
Ray — Architecture & Logic Reviewer
方案設計審查 · 開工簽核 · Veto 權 · Skeptic Check
Codex Plus (GPT-5.3)

為什麼用非 Claude 模型

避免 Claude 系統自我審查盲點（self-review bias）。Klay/Scout 用 Claude 設計 → Ray 用 Codex 審查 = maker-checker 不同模型。

Phase 3 職責（Major/Critical 任務）

  • 任務定義正確性（需求有沒有漏？目標是否明確？）
  • 方案架構風險（設計是否合理？邊界情況？）
  • Skeptic Check（必要輸出）：「如果不做這個呢？」「有更簡單的替代方案？」「假設 X 是錯的呢？」
  • 開工簽核：✅ 通過 / ❌ 打回 / ⚠️ 有條件通過
  • Veto 權：auth / payment / migration / production incident risk 可一票否決

Fallback Chain

  • Primary：codex CLI（ChatGPT Plus 訂閱）
  • Fallback 1：OpenRouter openai/gpt-5.3-codex
  • Fallback 2：Vick（Grok 只讀備援，不跑 lint/test）

Ray 不做什麼

  • 逐行 code review → Quill
  • lint / typecheck / pytest 執行 → Forge
  • 密鑰掃描 → Sentinel hook
  • 日常 post-implementation code review → Quill

KPI

北極星：事前審查攔截重工比例 ≥ 70%（被 Ray 攔下 vs 實作後才發現問題）

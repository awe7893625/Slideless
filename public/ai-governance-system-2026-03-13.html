Executive Summary

為什麼需要 Governance System

Rain AI Team 現有 28 個 Agent、7 個部門、15+ LaunchAgent 排程。每日執行數十個任務，但缺乏系統性的自動治理機制來確保架構一致性、規範遵守、任務可追溯性。

現有夜班審計覆蓋約 60-70% 的治理需求，但缺少 Config Diff、Review Compliance、Agent KPI 自動化、Self-Correction 四個關鍵能力。

8 個關鍵缺口

1. Config Diff System 缺失

沒有追蹤 agent config/prompt/model 變更的機制。Agent 角色或模型被改時，無人知道。

2. Agent KPI 自動計算 草稿

agent-kpi.md 只有 v1 定義，10 個 agent。無自動化計算，靠手動更新。

3. Dispatch Analytics 缺失

不知道哪個 agent 延遲最多、哪類任務失敗率高。無法回答「dispatch 有沒有效？」

4. Review Compliance Checker 缺失

無法偵測 Major 任務跳過 Ray 審查。Task model 甚至沒有 severity_level 欄位。

5. Cost-by-Agent Breakdown 部分

Token 成本有記在 task 上，但無分群分析、無預算警報。

6. Self-Correction Loop 缺失

發現問題後只有 Discord alert，沒有自動修復機制。

7. Centralized Audit API 缺失

夜班審計產 JSON cache，但沒有可查詢的 API endpoint 或 dashboard。

8. SKILL.md 升級 Enforcement 規則有，執行無

72h TTL 寫在 SOP，但沒有系統 enforce。逾期項目無自動提醒。

Skeptic Check（魔鬼代言人）

「如果根本不做 Governance System？」

現有系統已覆蓋 60-70%。真正缺的是 config diff（沒人追蹤 agent 配置變更）、review compliance（Major 任務可能跳審查）、role overlap detection（手動檢查易出錯）。

建議：不要一次建全部，分 3 phase 做。Phase 1 只要 1 天，立刻就有 80% 的價值。

Implementation Plan

三階段實施計畫

前置條件（Must-fix）

  • Kai task model 加 severity_level 欄位 — 否則 review compliance 無法做
  • 補齊 4 個 agent profile — quill / nautilus / echomarketer（flat→directory）、gooayeai（完全缺失）
  • Frozen agents 排除邏輯 — Milo/Sage 不列入缺失報告

Phase 1 — 獨立審計腳本（1 天）

建立 governance-audit.py，包含 4 個核心審計模組：

  • Agent 結構盤點（SKILL.md / persona.md / memory 完整性）
  • Config Diff（timestamp-based git diff）
  • Role Overlap Detection（責任重疊分析）
  • Review Compliance（任務等級 × 審查者矩陣對比）

輸出：JSON + Discord 通知。整合進 nightly pipeline 作為 Module H。

Phase 2 — Kai API 端點（第 2 週+）

將審計結果包裝成 API：

  • GET /api/governance/audit — 完整審計報告
  • GET /api/governance/agent-kpis — Agent KPI（利用現有 ScoreboardService + PostMortemService）
  • GET /api/governance/config-diff — 24h 配置變更
  • GET /api/governance/dispatch-analytics — Dispatch 分析

Phase 3 — Self-Correction Engine（第 4 週+）

Rules-based 自動修正，預設 alert-only，--auto-fix 需手動開啟：

  • agent_missing_memory → alert（不自動建空檔）
  • task_missing_postmortem → alert
  • scheduler_unregistered_job → alert（不自動註冊）
  • skill_upgrade_expired_72h → alert Quill
  • review_bypassed → alert Ray

所有修正記錄到 append-only JSONL，支援 --dry-run 預覽。

Architecture

系統架構設計

設計原則

  • 獨立 script 優先 — 不塞進 Kai main.py，避免後端膨脹
  • Alert-first — 先觀察再自動修，避免誤修
  • 漸進式 — Phase 1 用 1 天拿到 80% 價值
  • 向後相容 — 不破壞現有 night-shift / session review 機制

Config Diff 機制（Timestamp-based）

每次審計：讀取上次 last_audit_timestamp → git log --since={ts} --name-only -- .claude/agents/ .claude/team.json .claude/rules/ → 列出變更檔案 + diff → 寫入報告。

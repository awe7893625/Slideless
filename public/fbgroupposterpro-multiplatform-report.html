RAY PHASE 3

Ray 識別的問題：全部修復

Ray 審查給出「⚠️ 有條件通過」，識別 2 個 Blocker 和 3 個 Required 修改。所有項目在實作中均已解決。

  • B
    Blocker 1：event loop 阻塞

    Scout 原規格用 await handler()，會讓 blocking Selenium 呼叫跑在 event loop 上，癱瘓整個 aiohttp server。

    ✅ 修復：所有 _do_post_* 改為同步方法，透過 run_in_executor 呼叫

  • B
    Blocker 2：Instagram spike 未驗證

    execute_script 顯示隱藏 file input 的方法成功率不確定，需在真實帳號測試。

    ⚠️ 決策：實作 stub + guard，spike test 完成後才開放；Threads + X 可立即使用

  • R
    Required 1：PLATFORM_TIMEOUT 未套用

    timeout dict 定義了但 post_to_group 呼叫沒有包 wait_for，導致每個 group 都可能等到 1 小時全局 timeout。

    ✅ 修復：asyncio.wait_for(post_to_group(...), timeout=PLATFORM_TIMEOUT[platform])

  • R
    Required 2：Instagram 無圖片未 guard

    Instagram web UI 不支援純文字發文，無圖片時 Selenium 流程會卡住或報錯。

    ✅ 修復：posts.py dispatch 前 + _do_post_instagram 內部雙層 guard

  • R
    Required 3：fb_group_id 語義污染

    非 FB 平台不存在「fb_group_id」概念，強塞 URL 會讓資料模型語義混亂。

    ✅ 修復：新增 platform_entity_id 欄位存 X Community ID / Instagram handle 等

ARCHITECTURE

平台 Dispatch 架構：同步 Handler + run_in_executor

核心設計原則：所有 Selenium 操作都是 blocking I/O，必須透過 run_in_executor 推到 thread pool，讓 aiohttp event loop 保持回應。

Dispatch Flow

# selenium_engine.py — post_to_group() _POST_HANDLERS = { "fb": self._do_post, # 生產就緒 "threads": self._do_post_threads, # 中等風險 "x": self._do_post_x, # 高風險（長延遲） "instagram": self._do_post_instagram, # 極高風險（spike test 需先跑） } handler = _POST_HANDLERS.get(platform, self._do_post) # 關鍵：所有 handler 都是 def（同步），透過 executor 呼叫 result = await loop.run_in_executor( None, lambda: handler(session.driver, group_url, content, images or [], post_record_id) )

posts.py — Per-Group 呼叫（含 timeout + exception guard）

# posts.py — _run_post() group loop PLATFORM_TIMEOUT = {"fb": 300, "threads": 180, "x": 240, "instagram": 300} try: timeout = PLATFORM_TIMEOUT.get(platform, 300) result = await asyncio.wait_for( selenium_engine.post_to_group(..., platform=platform), timeout=timeout, ) except asyncio.TimeoutError: result = {"success": False, "error": f"Timeout after {timeout}s"} except Exception as exc: result = {"success": False, "error": str(exc)}
PLATFORM ASSESSMENT

各平台評估：封帳風險 × Timeout × 建議

Selenium 自動化對各平台的難度和風險差異很大。以下是實際可行性評估。

🔵 Facebook
300s
原有引擎，已在生產環境驗證。undetected-chromedriver + navigator.webdriver 偽裝已套用。群組 compose UI 多個 selector fallback。
封帳風險：低
⚫ Threads
180s
Meta 2023 年推出，UI 仍在迭代。aria-label 'New thread' compose button + contenteditable text area。UI 改版頻率中等，需監控 selector 有效性。
封帳風險：中
🔷 X (Twitter)
240s
data-testid="tweetTextarea_0" 較穩定。個人 timeline + Community URL 雙模式支援。輸入延遲加長（0.05-0.15s/char）降低被偵測機率。
封帳風險：高
🌈 Instagram
300s
execute_script 顯示隱藏 file input 方法需 spike test 確認。3 步驟流程（選圖→裁切→說明）。必填圖片。建議先在測試帳號跑一次再開放。
封帳風險：極高（spike 待驗）

⚠️ Instagram Spike Test 步驟

  1. 用真實 Instagram 帳號啟動 browser session
  2. 準備一張本機圖片（JPG/PNG）
  3. 呼叫 POST /api/posts，指定 platform=instagram 的群組，帶入 images 路徑
  4. 觀察是否能成功上傳圖片並完成發文流程
  5. 成功 → 可開放使用；失敗 → 需研究 Instagram CSP 變化

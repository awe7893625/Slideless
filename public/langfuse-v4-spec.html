必做 1 · 最高優先

Real-time Instrumentation：完整執行週期 Span 化

現在是 force_complete() 後補寫 trace。v4 要在每個關鍵 lifecycle event 當下寫 span，讓長任務中途可見、失敗可即時定位。
現狀（v3）

任務完成後補記

# 現在的流程 task.status = "completed" langfuse_service.emit_task_complete(task, agent_steps) # 問題： # - 中途失敗 → trace 可能遺失 # - 長任務執行中完全黑盒 # - 無法 dhoc 定位卡點
v4 目標

每個 lifecycle event 即時寫 span

task.start 任務接收，建立 Trace
agent.dispatch subagent 被派遣
llm.inference LLM 呼叫前後
tool.execute Bash/Write/Grep 執行
retrieval.query Memory/RAG 查詢
retry / fallback 重試與降級
task.complete 最終結果 + 成本匯總
實作規格 — langfuse_service.py v3 升級

新增 emit_span() 方法，支援 start/end 分開呼叫

# v4 新增 API class LangFuseService: def emit_span_start( self, trace_id: str, span_id: str, span_type: str, # llm | tool | retrieval | agent name: str, model: str = None, input_summary: str = None, metadata: dict = None, ) -> None: """任務中途 span 開始，fire-and-forget""" ... def emit_span_end( self, span_id: str, output_summary: str = None, token_input: int = 0, token_output: int = 0, error: str = None, # 失敗時填入 ) -> None: """span 結束，計算 latency + cost""" ... # 使用方式（llm_service.py 整合） span_id = f"{task_id}-llm-{uuid4()}" langfuse.emit_span_start(trace_id, span_id, "llm", "claude.inference") result = await llm.complete(prompt) langfuse.emit_span_end(span_id, result.text, result.usage.input, result.usage.output)
整合目標：llm_service.py

LLM 呼叫自動記錄

在 llm_service.py 的統一 call() 方法加 span，所有 agent 呼叫自動有 trace，不需要每個 agent 手動添加

整合目標：tool_executor.py

工具執行 span

Bash / Write / Grep 等工具執行前後，記錄 tool_name、latency_ms、成功/失敗。tool failure rate 可從此統計

整合目標：memory_service.py

Retrieval span

Memory/RAG 查詢時記錄 retrieval_source、query、hit_count。偵測高延遲查詢與低命中率問題

必做 2

Task-type Evaluation Framework：多維度品質評分

不是單一 0-1 分數。依任務類型定義不同評分維度與權重，讓 Quality 可量化、可比較、可回歸偵測。
評分維度（4 軸）

所有任務類型都從這 4 個維度評

維度定義評分方式爬蟲任務報告任務Coding秘書型
Correctness事實正確 / 資料完整LLM-as-Judge + 人工抽驗40%35%40%20%
Completeness任務需求是否全部達成Checklist + LLM 驗證35%30%35%40%
Format格式符合規格 / 可直接使用Schema 驗證 / 規格比對15%20%20%30%
Efficiency成本 + 時間是否合理與同類任務基線比較10%15%5%10%
LLM-as-Judge 設計

三種評分器

1
Pairwise Judge（比較型）
「這次任務結果」vs「Golden Set 標準答案」，判斷哪個更好，輸出 win/tie/lose + 理由
2
Criteria Judge（標準型）
給 rubric（評分細則），逐項打分 0-10，最後加權彙總。每個任務類型各有一份 rubric
3
Schema Validator（格式型）
工具輸出、爬蟲資料、API 回應 → 用 Pydantic schema 驗證完整性，不需要 LLM，精確快速
Langfuse Scores API 整合

評分結果寫回 trace

# 評分結果送回 Langfuse POST /api/public/scores { "traceId": task_id, "name": "quality_score", "value": 0.82, "comment": "correctness:8 completeness:9 format:7", "dataType": "NUMERIC" } # Kai task 同步 task.quality_score = 0.82 task.quality_comment = "..." task.evaluated_at = now()

評分結果同步回 Kai task DB，中控台可以直接看任何任務的品質分數。

Regression Detection

品質退化自動偵測：連續 3 個同類任務分數 < 基線 → 建立告警

# regression_detector.py（夜班 Module R） def detect_regression(task_type: str, window: int = 7) -> dict: recent = get_recent_scores(task_type, limit=10) baseline = get_baseline(task_type) # 前 30 天 p50 if recent.p50 < baseline * 0.85: # 退化超過 15% create_kai_task("[Regression] 品質退化偵測", task_type) alert_discord(f"⚠️ {task_type} p50 品質從 {baseline:.2f} 降到 {recent.p50:.2f}")

Data Governance：密鑰、報告分層、Trace Redaction

v3 在公開報告出現 API key fragments。v4 建立完整的資料治理規範：什麼可以公開、什麼要遮蔽、trace 資料的隱私邊界。
立即執行（已修復）

API Key 遮蔽

  • 公開報告禁止出現任何 key / secret
  • Langfuse host URL 加入遮蔽清單
  • v3 報告已移除 key fragments
  • Sentinel hook 掃描規則新增 pk-lf / sk-lf 模式
報告分層規範

Internal / External 分開

  • Internal 含 trace 詳情、cost breakdown、model 名稱
  • Public/Deck 只含聚合指標、無 key、無個人資料
  • Deck 報告統一用 Public tier，不含 raw trace
  • Kai 中控台 = Internal tier（需登入）
Trace Redaction 規格

Task 資料遮蔽規則

  • input/output 含 email → 自動替換為 [EMAIL]
  • 含 password / token 的字串 → [REDACTED]
  • SQL 連線字串 → [DB_URL]
  • 遮蔽在 emit_task_complete() 前處理
Sentinel Hook 新規則

新增 Langfuse key 掃描模式

必做 6

Span Storage Strategy：短期 JSON，長期正規化

agent_steps JSON 欄位短期 OK，但當你要做跨任務統計、tool failure heatmap、retrieval 分析時就會很痛。v4 規劃遷移路徑。
現狀（短期保留）

agent_steps JSON Column

# tasks table agent_steps = Column(JSON) # 存入格式 [{ "agent_name": "max", "span_type": "llm", "model": "claude-opus-4-6", "latency_ms": 3500, "cost_usd": 0.035 }] # 優點：快，不破壞現有 schema # 缺點：無法 indexed 查詢
長期目標（v4 Phase 2）

獨立 task_spans 表

# 新增 task_spans table CREATE TABLE task_spans ( id UUID PRIMARY KEY, task_id UUID REFERENCES tasks(id), span_type VARCHAR(16), -- llm/tool/retrieval/agent agent_name VARCHAR(50), model VARCHAR(80), tool_name VARCHAR(80), latency_ms INTEGER, cost_usd DECIMAL(10,6), token_input INTEGER, token_output INTEGER, started_at TIMESTAMP, ended_at TIMESTAMP, error TEXT -- null = success ); -- Index for analytics CREATE INDEX ON task_spans (span_type, agent_name); CREATE INDEX ON task_spans (tool_name) WHERE span_type = 'tool';
遷移策略

三步驟平滑遷移（無停機）

Step 1
雙寫過渡

force_complete() 同時寫入 agent_steps JSON 和新 task_spans table。兩者共存 4 週，驗證數據一致

Step 2
讀取切換

metrics API 和 agent 統計改從 task_spans 讀取。確認結果一致後停止讀 agent_steps JSON

Step 3
舊欄位廢棄

停止寫入 agent_steps JSON，標記為 deprecated，計畫在 v5 schema migration 中刪除

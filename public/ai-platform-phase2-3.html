Phase 3 — Ray 審查：BLOCKING 問題 必須修復才能開工
以下 4 個問題必須在實作前解決，否則架構存在根本缺陷
🚨
4 個 BLOCKING 問題
實作前必須解決
1

Running task timeout watchdog 缺失

任務狀態機中，running 狀態缺乏超時保護。一旦 Worker 崩潰或斷線，任務將永遠卡在 running 狀態，無法重試或標記失敗。

修復方案：tasks 表加 timeout_seconds 欄位 + APScheduler job（每分鐘掃描 running 中超時任務 → 標記 timeout）
2

/api/worker/poll 無 Rate Limiting

Worker SDK 輪詢端點沒有請求頻率限制。若有 SDK 設定錯誤（輪詢間隔過短或無限迴圈），將直接打爆 DB 連線池，影響整個平台。

修復方案：Redis rate limit，每個 provider_id 最多 12 req/min，超過返回 HTTP 429
3

Package Builder 重啟後 build_status 卡在 building

平台重啟後，若有任務在 building 狀態，系統會認為「打包進行中」，使用者看到永久 building 的假象，且該模板無法重新觸發打包。

修復方案：平台啟動時執行 SQL UPDATE templates SET build_status='failed' WHERE build_status='building'
4

Matching 演算法不排除 busy providers

當前 matching 邏輯只依技能匹配 Provider，沒有檢查 Provider 是否已有進行中的任務。一個 Provider 可能同時接到多個任務，超出其處理能力。

修復方案：matching SQL 的 WHERE 條件加上：NOT EXISTS (SELECT 1 FROM tasks WHERE provider_id = p.id AND status IN ('assigned','running'))

Ray Skeptic Check
魔鬼代言人視角 — 必要輸出，不可省略
❓ 模板可以不依賴平台嗎？
Ray 意見：模板本身可以完全獨立運作，不依賴平台存在。建議：先建模板驗證使用者價值，再加平台層。這樣萬一平台設計調整，模板仍可交付。
❓ 雙重 Auth 是否過於複雜？
Ray 意見：設計正確。JWT vs opaque token 的分離在使用場景上是必要的。bcrypt + Redis 快取是必要條件，不是 nice-to-have，缺少 Redis 快取會讓 /health 和 poll 端點變慢。
❓ Matching 演算法的 busy-provider 排除真的是 P0？
Ray 意見：是的，這是 P0，不是 P2。不是「效能問題」，而是「正確性問題」。Provider 同時執行多個任務不僅超載，也會讓任務結果混淆。必須列為 BLOCKING 問題修復。

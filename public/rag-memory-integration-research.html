01執行摘要

核心結論

我們已經擁有 80% 的記憶基礎設施，包括 vector DB（SQLite + OpenAI embeddings）、 對話記憶、摘要壓縮、RAG 索引、Obsidian 同步、task reflections 等。 不需要重做，只需要：

  • 補缺口：記憶分層（user/project/task/system/knowledge）的 metadata schema
  • 接管線：把分散的記憶源統一到一條 retrieval pipeline
  • 治理規則：什麼值得存、誰能寫、何時淘汰
  • 整合 Obsidian：從「鏡像層」升級為「可檢索的知識層」

05三套架構方案

方案 A：最小改動版 推薦先做

原則：不新增任何基礎設施，只補缺口

做什麼 1. 驗證 task_reflections FTS5 運作 + 補齊缺漏欄位
2. 統一 memory metadata schema（type / project / source / created_at）
3. 把 Claude memory .md 檔加入現有 LightVectorStore 索引
4. 建立 /api/memory/unified-search — 同時搜 vector DB + FTS5 + Claude memory
不做什麼不新增 ChromaDB，不動 Obsidian，不改現有架構
工時2 天（Max 後端）
成本$0（用現有 OpenAI embedding）
Token 節省20-30%（統一搜尋減少重複載入）
破壞度極低（只新增 API + 表）

07記憶治理規則

什麼值得存 vs 不值得存

值得存不值得存
已決策事項（含原因）
長期偏好（驗證過的）
專案關鍵背景
重複使用的研究結論
Agent handoff 必要資訊
錯誤教訓 + 修正紀錄
API / 服務 credential 位置 		無意義寒暄
一次性指令噪音
程式碼可直接推導的重複內容
git log 可查的歷史
超過 180 天未存取的暫時記錄
主觀推測（未標註 confidence）
debug log 原始輸出

寫入規則

  • 誰寫：統一由 Kai memory service 寫入 vector DB；Claude Code 寫自己的 .md；Obsidian 由 obsidian-team 管理
  • 何時寫：任務結束時（非中途）+ 中途 checkpoint + 人工審核後
  • 品質門檻：必須有 source trace + confidence（若為推論）+ >100 字內容
  • 防污染：hash dedup → 同內容不重複存；confidence < 0.5 不進 vector DB

檢索規則

  • Top-K：預設 5 條，Major+ 任務 10 條
  • 分層檢索：先 project scope → 再 global scope
  • 加權：最近 30 天 ×1.5 權重；importance=high ×2.0
  • Agent 權限：所有 agent 可讀全部；credential 類只有 Knox 可讀

淘汰規則

  • 90 天未被檢索命中 → 標記 stale
  • 180 天仍 stale → archive（從 vector DB 移除，Obsidian 保留）
  • 月度清理：Orion 每月掃描 → 報告重複 / 過期 / 低品質記憶

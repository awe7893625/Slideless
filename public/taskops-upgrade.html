1 問題診斷 — 為什麼需要升級

當前系統的結構性缺陷 + GitHub 森林系設計的核心洞見

當前系統的 5 個結構性缺陷

1
狀態機過於簡單：只有 3 個狀態（pending → executing → done），無計畫門控， 任何任務都可以直接跳到執行，缺乏關鍵節點的決策管控。
2
沒有正式審批流程：夜班任務直接執行，Major/Critical 任務缺乏人工確認機制， 高風險操作可能在 Rain 不知情的情況下執行。
3
技能管理混亂：110+ Skill 散落在各 SKILL.md 文件，無機器可讀的 registry， 也無法追蹤 Skill 的狀態（廢棄/活躍）、成本級別和負責 Agent。
4
任務路由依賴文字：分類依賴 CLAUDE.md 文字規則表，沒有自動分類引擎， 無法以程式化方式追蹤 dispatch_rate 和確保路由一致性。
5
執行隔離不完整：沒有狀態轉換審計日誌，無法追溯任務的完整生命周期， 也無法偵測並發衝突或異常狀態轉換。

GitHub 森林系設計的核心洞見（吸收而非複製）

🧠
CLAUDE.md as project brain：我們已有完整的 CLAUDE.md 體系， Phase 1 的目標是加強治理而非重建，在既有框架上疊加狀態機和審計能力。
🌿
每任務獨立 worktree：Worktrunk 已在使用，Phase 1 補正式生命周期管理， 記錄 worktree_branch 欄位，Phase 2 再引入 Conductor daemon 完成閉環。
📋
計畫先行，再執行：PLANNED → APPROVED 的計畫門控是最高優先事項。 Major/Critical 任務必須提交結構化計畫（JSON）並通過審批才能進入執行。
🔀
PR 作為交付物：任務完成後提交 PR，進入 REVIEW_REQUIRED 狀態， 觸發 Quill → Forge → Ray 三層審查。Phase 3 正式接入 GitHub webhook。

3 個高嚴重度風險（H 級）— 問題 vs 解決方案

H-1 SQLite 並發 Write-Lock 風險
問題

並行任務狀態轉換會觸發 SQLite write-lock，導致任務卡住或資料不一致， 夜班高並發場景尤為危險。

✅ 採納解決方案

已啟用 WAL（Write-Ahead Logging）mode + NORMAL synchronous 模式， 大幅提升並發讀寫能力，消除大部分 write-lock 場景。

H-2 WAITING_APPROVAL 死鎖 — 夜班佇列堵塞
問題

夜班期間 Rain 無法回應審批請求，若 Major 任務卡在審批狀態， 後續任務佇列全數堵塞，夜班工作無法推進。

✅ 採納解決方案

加入 8h/24h 逾時機制：Major 任務 8 小時後 Klay 可代為核准； Critical 任務 24 小時後通知 Rain 但不強制核准，確保佇列不死鎖。

H-3 GitHub 優先入口假設錯誤
問題

原設計將 GitHub 列為主要入口，但 Rain Team 所有任務都來自 Kai Web UI / Claude Code / Telegram，而非 GitHub Issues。

✅ 採納解決方案

Kai / Claude Code 為主入口，GitHub 降為 Phase 3 次要入口， 僅作為 PR 交付物的引用來源，不作為任務建立觸發器。

🔍 Ray 的 Skeptic Check — 魔鬼代言人挑戰
「3 個小 fix 就能達到 80% 的價值：加 PLANNED 狀態 + 加審批欄位 + 加 Skill JSON。 你確定需要建立整套 TaskOps 框架嗎？」

Klay 的回應：採納觀點，但選擇建立完整架構基礎。 3 個小 fix 確實能解決眼前問題，但 6 個月後當我們需要 Conductor daemon、 PR Bridge 和多模型審查時，還是要回來重構，屆時的技術債代價更高。 Phase 1 已將 scope 控制在最小可行範圍（不含 daemon、不含 GitHub webhook）， 是正確的分期策略，不是過度工程。

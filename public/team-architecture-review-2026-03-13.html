根因：Claude Code session 中 Klay 直接實作而不 dispatch，因為 dispatch 到 subagent 需要額外 context 準備和等待時間。

根因：Ray 用 Codex Plus CLI，執行環境不穩定；Scout 從未被 Klay dispatch 過。3-Phase Protocol 是設計良好但從未落地的流程。

4. 改善建議

1
R-01: 強制 Dispatch — Klay 不寫實作程式碼 P0
修改 Klay 的行為規則：feature/bug_fix 類任務必須透過 Agent tool dispatch 給 Max/Rex/John/Iris，Klay 只做規劃、協調、報告。
實作方式：在 CLAUDE.md 加入硬規則「Klay 看到 feature/bugfix 任務 → 必須 dispatch，禁止自行實作」。
預期效果：Klay 負載降至 30% 以下，Core 團隊利用率從 9% 升至 50%+。
2
R-02: 簡化審查管線 — 從 5 層降為 3 層 P1
當前 5 層審查全部斷裂。建議精簡為：
Layer 1: Quill code review（Claude Code subagent，自動追蹤到看板）
Layer 2: Forge CI（整合 lint+test+build 為一個自動 step）
Layer 3: Ray design review（僅 Critical 任務，修復 Codex 環境後）
取消 Sentinel 獨立任務追蹤（保留 git hook 但不上看板）、取消 Probe+Echo 常規 UI 測試（保留手動觸發）。
3
R-03: 重新定位 Noel — 從「備用助理」到「夜班 + 外部協調」 P1
Noel 的「備用助理」定位太弱，導致永遠不會被選中。建議重新定位：
1. 夜班主力執行者：Night Shift 排程任務由 Noel 執行（非 Klay）
2. External Worker 管理：Spider/Lynx/Nautilus/Buzz 的任務由 Noel dispatch 並追蹤
3. 移除 TW accounting：此職責完全由 Tax 承擔，Noel 不需要
觸發條件：夜班任務 / 外部爬蟲任務 / Rain 個人事務
4
R-04: 降級 Jarvis — 從指揮鏈移除 P1
Jarvis 在指揮鏈中位於 Rain 和 Klay 之間（Rain → Jarvis → Klay），但實際上 Rain 直接和 Klay 對話。建議：
1. 移除指揮鏈中間層：Rain → Klay（直接）
2. Jarvis 降為工具：保留 Email 寄送功能（若 Kai API 不足時的備用），但不作為獨立 Agent
3. Night Shift 排程：由 Kai cron 系統直接觸發，不經 Jarvis
預期效果：減少一個指揮鏈節點，降低組織複雜度。
5
R-05: 合併冗餘角色 — 減少 Agent 總數 P2
建議合併：
- Milo → 刪除：Spider/Lynx 直接向 Noel 報告，Milo 的 ETL 工作由 Max 處理
- Knox + Sentinel → 合併為 Security Lead：統一管理密鑰 + 安全掃描
- Sage → 合併入 Knox：account lifecycle 不需要獨立 agent
- Nova → 合併入 Klay：Deck 報告目前由 Klay 直接生成
合併後：32 agents → ~26 agents，減少 6 個零活動角色。
6
R-06: 清理看板 — 刪除空殼任務 + 追蹤改善 P2
1. 清除 8 個 "Terminal Session" pending 任務
2. Subagent 執行（Quill review、Forge CI）需回報到 Kai 看板，否則看板數據不完整
3. 加入 agent utilization dashboard（每個 agent 的任務完成率、平均耗時）

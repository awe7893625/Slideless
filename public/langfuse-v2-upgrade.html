01 · GAP ANALYSIS

缺口分析：工程師說得對的地方

原版 6/10 — 方向正確，但缺少工程細節。以下是確認屬實的缺口

✗

缺 Cost 換算 已修復

有 token 數（token_input / token_output）但沒換算成 USD。不同 model 費率差異最高 100x（Opus $75/M vs DeepSeek $1.1/M），沒有 cost_usd 就無法做成本管控。

✗

缺 Latency Breakdown 已修復

agent_steps 有 latency_ms 欄位但沒有彙整到 trace metadata。無法知道多 agent 任務中哪個 subagent 最慢，也就無法針對性優化。

✗

缺 Alert 機制 已修復

高成本或高延遲任務沒有即時通知。一個任務花了 $0.50 或跑了 3 分鐘，沒有任何警報 — 這在 production 系統是不可接受的。

✓

確認不修（本次）：Real-time Span Phase 2

我們的 trace 是「任務完成後送 summary」而非「LLM call 即時打 span」。這是架構限制，需要在每個 llm_service.py call 點注入 instrumentation — 這是 Phase 2 工作，與本次目標不同。

✓

確認不修（本次）：Evaluation Pipeline Phase 3

LLM judge 評分需要先累積 2-4 週數據集才有意義。目前 7 traces，量太少。這是 Phase 3 工作。

02 · NEW FEATURES

本次新增：3 項功能

全部在 langfuse_service.py v2 實作，不修改其他模組

① Model Cost Table

20+ 模型費率表，Input / Output 分開計價，精確到 $0.000001。
含模糊匹配（去掉日期後綴自動對應），claude-sonnet-4-6-20260301 自動匹配到 claude-sonnet-4-6。

寫入兩個地方：① task.cost_usd DB 欄位（永久儲存）② LangFuse trace metadata（可視化）

② Latency Breakdown

agent_steps 中每個 subagent 的 latency_ms 被彙整為 latency_breakdown_ms dict，寫入 trace metadata。

"latency_breakdown_ms": { "max": 3500, "john": 2100 }

③ Discord Alert

任務完成後自動判斷是否觸發告警（fire-and-forget，不阻塞）：

  • cost_usd ≥ $0.10 → Discord 通知
  • duration ≥ 60s → Discord 通知

Discord 訊息包含：任務名稱、agent、model、token 總數、cost、latency breakdown。

Discord Alert 格式

🔍 LangFuse Alert — 💰 高成本 $0.1200
任務: 重構商品管理模組
Agent: max | Model: claude-opus-4-6
Tokens: 8,500 | Cost: $0.1200 USD
Duration: 45s
Latency breakdown: max: 28000ms | john: 17000ms

06 · HONEST ASSESSMENT

誠實評估：仍然缺什麼

8/10 不是 10/10 — 以下是真實存在的差距

Real-time Span（最大差距）

我們現在：任務完成後送一條 summary trace。

真正企業級：每個 LLM API call 即時打 span — 在 llm_service.py 的每次 client.messages.create() 調用前後插入 instrumentation。

這是 Phase 2 的工作，需要修改 llm_service 所有路徑，影響面廣。

Evaluation Pipeline（Phase 3）

LLM judge 評分 / 準確度分析需要：

① 足夠數據集（建議 200+ traces）
② 定義評分標準（accuracy / relevance / hallucination）
③ 定期批次評估 pipeline

目前 7 traces，2-4 週後數據量才夠啟動。

Dashboard 聚合（近期可做）

LangFuse 已有 traces，但沒有：

• 每天 cost_usd 總計
• 哪個 agent 最貴
• 哪種 task_type 最慢

可用 /api/public/traces 定期跑統計 → 市場晨報中加一欄。1 週後可做。

Trace 覆蓋率

目前只有 force_complete() 路徑會送 trace。

正常 claim → start → complete 路徑、失敗任務（FAILED 狀態）、retry 任務都還沒有 trace。

估計實際覆蓋率：~40% 的任務有 trace。

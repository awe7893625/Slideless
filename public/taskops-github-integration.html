BLOCKED 狀態
任務被外部因素阻擋（非依賴關係），例如：等待 Rain 確認、等待第三方 API key、等待客戶回覆。

trigger：block (RUNNING → BLOCKED)
trigger：unblock (BLOCKED → RUNNING)
⏳
WAITING_DEPENDENCY 狀態
等待其他任務完成的依賴關係。前置任務完成後自動觸發 dependency_resolved，回到 APPROVED 重新執行。

trigger：dependency_pending
trigger：dependency_resolved

非阻塞設計保證

🔒
try/except 包裝策略
所有 GitHub API 呼叫均以 try/except 包裝。GitHub 連線超時、Rate Limit、Token 失效等任何錯誤，均只記錄 log，不影響 Kai Task Board 主流程。
📡
WAL Mode SQLite
SQLite 使用 WAL（Write-Ahead Logging）模式，支援多 writer 並發。確保 TaskOpsBoard 前端讀取與後端狀態更新不互相阻塞。
🌊
Noel Module G
夜班新增 Module G：定期輪詢 /gh/poll，偵測 GitHub 上直接開立的 Issue 並自動建立對應 Kai task，實現雙向同步。

GitHub Labels 設計（28 個）

完整的 GitHub Label 體系，按 prefix 分組設計，支援未來無限擴充新 agent 類型和工作流程，不需修改任何後端程式碼。

State Labels（9）— 對應 9-State Machine

taskops:pending taskops:planned taskops:approved taskops:running taskops:blocked taskops:waiting taskops:review taskops:done taskops:failed taskops:cancelled

Category Labels（6）— 任務類型分類

Severity Labels（4）— 對應任務分級

severity:tiny severity:standard severity:major severity:critical

Base Label（1）

taskops 所有同步任務都有這個 label，方便 filter
🔧
Label 設計原則
按 prefix 分組：taskops:* / severity:* / taskops（base）

新增 agent 或工作流程，直接新增對應 label 即可，後端 TASK_ROUTING 和 State Machine 無需修改。
🔄
自動 Label 管理
首次同步時若 label 不存在會自動建立。Label 顏色按語義設定（綠=完成、紅=危險、橘=進行中），與 GitHub 原生 UI 視覺一致。

Section 10

對未來 AI 團隊擴張的設計考量

Rain 明確要求：「不要只以我們現在的陣容來說，我們之後的陣容可以一直增強。」 TaskOps 的每個設計決策都以「未來可以無限擴充」為前提。

🔢
State Machine 可擴展
目前 9 個狀態。Phase 4 計畫支援 11+ 個狀態（預留 ESCALATED、DELEGATED、PAUSED）。狀態轉換以字典定義，新增一個狀態只需加一個 key-value，不需修改 transition 邏輯。
🏷️
GitHub Labels 按 Prefix 分組
新增 agent 類型直接新增對應 label（如 agent:nova / agent:scout），GitHub API 會自動建立。不需修改任何後端程式碼，標籤系統天然支援無限擴充。
👥
AUTHORIZED_APPROVERS 可擴充
目前審批人：{"klay", "rain"}。未來加入新的高級 Agent，直接 add 到 set 即可。APPROVED 邏輯不需改動，自動支援新審批人。
🗺️
TASK_ROUTING 關鍵字擴充
目前 6 個 category（quick/standard/deep_research/infra/review/release）。新增 category 只需在 keyword map 加一行，routing 邏輯自動識別，dispatch 流程不受影響。
⚡
dispatched_to 支援並行
dispatched_to 欄位支援逗號分隔多個 agent（如 "max,john"），未來雙人並行甚至三人並行任務天然支援，Kanban 看板一張 card 可顯示多個 agent badge。
🏗️
WAL Mode 支援多 Writer
SQLite WAL 模式支援多個 agent 同時寫入任務進度，不互相阻塞。未來即使 20+ agent 同時回報進度，資料庫也不會成為瓶頸。（估計可支援 ~50 concurrent writers）
🔍
Skill Registry Sprawl 偵測
Sprawl threshold = 150，超過自動發 Discord 警告。Noel Module N（Phase 3）夜班每晚自動驗證 registry 完整性，確保 skill 數量增長不失控。
🌐
GitHub 作為統一審計軌跡
無論未來加入多少 agent，GitHub Issue 永遠是最終的審計軌跡。每個 state 轉換、每次進度回報都有 timestamp + agent 簽名，Rain 可隨時翻查任何任務的完整歷程。

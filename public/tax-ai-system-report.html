第 2 章 — RAG 流程

檢索增強生成（Retrieval-Augmented Generation）— 混合式檢索架構 7 步驟

架構流程圖

Question Input ↓ Step 1: 元數據過濾 (10ms) ├─ tax_type IN [...] AND status='active' ├─ effective_date ≤ year-12-31 └─ Output: ~20% 相關 chunk ↓ Step 2: BM25 關鍵字搜尋 (50ms) ├─ rank_bm25 Python 庫 └─ Output: Top 20 關鍵字匹配 ↓ Step 3: 向量語義搜尋 (100ms) ├─ LightVectorStore (SQLite) └─ Output: Top 20 語意相近 ↓ Step 4: 合併去重 └─ Union by chunk_id → ~30-40 候選 ↓ Step 5: 重排序 (200-500ms) ├─ cross-encoder/ms-marco-MiniLM-L-6-v2 └─ Output: Top 5-8 chunk ↓ Step 6: 法源層級確保 └─ ≥1 母法 + ≥1 解釋令/函釋 + ≥1 QA ↓ Step 7: 法律基礎驗證 └─ confidence: high/medium/low

效能指標

  • 總延遲：500ms - 2s
  • 元數據篩選：10ms
  • BM25 搜尋：50ms
  • 向量搜尋：100ms
  • 重排序：200-500ms

成本估算

  • 單次查詢成本：$0.05 - $0.15
  • 模型調用：Claude Opus
  • 降級策略：若 sentence-transformers 不可用，用 LLM 評分

第 3 章 — 三步解題規則

規則 1：四選項逐一排除法

觸發條件：題目包含「何者正確」或「何者錯誤」

  • 提取四選項 (A)、(B)、(C)、(D)
  • 對每選項進行二元判斷：✅ 正確 or ❌ 錯誤
  • 分析判斷理由（參照 RAG 檢索的法源）
  • 根據題目要求確定答案
  • 輸出答案 + 逐項分析

規則 2：計算題調用 Calculator API

觸發條件：題目包含數字（金額/稅率/比率/期間）

  • 識別計算類型（所得稅/營利稅/營業稅等）
  • 提取參數（金額、稅率、年份等）
  • 呼叫 POST /api/tax-calc/calculate（year 參數必帶）
  • 驗證結果合理性（負稅額立即標記）
  • ⛔ 禁止 LLM 心算

規則 3：程序題查 RAG 條文

觸發條件：題目包含「期限/期間/日期/幾年內/幾日內」

  • 強制查 RAG（禁止背誦，台灣稅法每年可能修訂）
  • 建立查詢：問題類型 + 關鍵法律範圍
  • 篩選最相關的一條（通常是母法）
  • 提取具體的『數字』（如：5年、30日）
  • 應用於題目情景
✓ 基準驗證

112 年會計師考試：25/25 (100%) — 所有題型都被三步規則覆蓋

第 4 章 — 諮詢流程（使用者面向）

用戶旅程 — 6 階段

Phase 1：問題輸入

  • 通道：網站表單、Telegram Bot、API
  • 驗證：問題必須 >5 字，不能純數字
  • 欄位：question (5-1000 字)、context (可選)、tax_type (可選)、year (可選)

Phase 2：路由決策

if 包含計算關鍵詞: route='calculator' elif 選擇題: route='exam_evaluator' else: route='rag' # 預設法律查詢

Phase 3 & 4：檢索 & 推理

  • RAG 路由：混合檢索（BM25 + 向量 + 重排）→ Top 5-8 條文
  • 計算路由：呼叫計算器 API → 結果 + 步驟 + 驗證
  • 考試路由：四選項排除 → 答案 + 分析
  • 推理層：Claude Opus（法律判斷 L7 → 會計判斷 L8 → 計算驗證 L9 → 風險檢查 L10）
Phase 5：輸出格式化（強制 6 欄）
  • conclusion — YES/NO 或直接答案
  • legal_basis — 法源清單（law_name, article, hierarchy, excerpt, url）
  • reasoning — 邏輯解釋
  • accounting_treatment — 會計帳務處理
  • tax_treatment — 稅務上的處理
  • risks — 潛在補稅/罰款風險
【強制】法律免責聲明

AI 生成的參考資料，不是專業稅務建議。重要決策請諮詢持證 CPA/稅務律師。

關鍵設計決策 & 風險

Critical Blockers（外部上線前必須解決）

⚠️ C-1：法律免責聲明強制包含

  • 文本：【AI 生成的參考資料。不構成專業稅務建議。重要決策請諮詢持證 CPA 或稅務律師。】
  • 實施：輸出模板中強制包含，不能跳過、摺疊或移除
  • 法律基礎：台灣《會計師法》第 39 條（未經授權提供稅務諮詢有法律責任）
  • 狀態：此次報告已更正 TaxQuery 模型

⚠️ C-2：稅務年度自動解析

  • 需求：Router 必須強制提取『適用稅務年度』作為必備事實
  • 邏輯：若使用者未指定年份，必須：(A) 推導自問題內容，或 (B) 預設為當年度並明確標註
  • 輸出格式：每回答顯示『使用法律版本：xx 年』
  • 原因：台灣稅法每年變動，同一問題用不同年份法律答案可能完全不同
  • 狀態：此次報告已新增 year_context 欄位到 TaxQuery

High Priority（上線前檢查）

  • C-3 法源完整性驗證：每回答確保 ≥1 母法 + ≥1 解釋令/函釋 + ≥1 QA（law.moj + NTBT + 財政部）
  • C-4 計算器回歸測試：所得稅/營利稅/營業稅/房地合一/遺贈稅 完整測試
  • SQLite 並發處理：啟用 WAL mode + Redis 查詢快取
  • RAG 語料庫完整性：爬蟲確保每月更新，關鍵法律不漏網

Medium Priority

  • C-5 OBU/DBU 元數據：新增 jurisdiction + treaty_applicability 欄位；Router 添加 OBU/DBU/外籍專才/雙重課稅 關鍵詞觸發
  • 版本切換複雜性：跨年度查詢時法律版本自動切換測試
  • 重排序單點故障：降級方案（若 cross-encoder 不可用，用 LLM 快速評分）

現在 vs 頂配 — 差在哪裡

現狀問題
各模型一起 Review
Claude、GPT、Gemini 各說各的，統整後沒有「權威判決」。結果像委員會決策 — 緩慢、模糊、沒人負責對錯。
✗ 無 verdict 責任人
✗ 模型衝突意見無法裁決
✗ 沒有人負責最終對錯
頂配架構
Supreme Judge 閉環
多模型並行 review，Opus 做最終 verdict 裁決。系統有明確的責任終點，每個決策都有問責鏈。
✓ Opus = 唯一最終決策者
✓ 模型衝突由 Opus 一票裁決
✓ Accept / Revise / Reject 三態明確

根本差異：責任歸屬。現在的系統沒有人真正負責「這個 code 能不能上 production」。頂配架構讓 Opus 成為那個負責人 — 它說 Accept，才能上線；它說 Reject，不管有多少其他 reviewer 通過也不行。

Opus 的 三個正確角色

最重要
Supreme Judge — 最終裁決者
這是 Opus 最核心的存在理由。不是在 review 池裡多一個意見，而是整個 review 鏈路的終點。
  • 最終 verdict 唯一決策權（accept / revise / reject）
  • 處理模型衝突 — Sonnet 說 OK、Gemini 說有問題，Opus 仲裁
  • 判斷是否可上 production，沒有其他模型可以覆蓋
  • 輸出結構化 JSON，機器可讀，直接接 Auto Fix Loop
架構審計
Architecture Auditor
大型 feature / refactor / infra / agent 系統時觸發，不是每次 review 都用。
  • Scalability — 能撐住 10x 流量嗎？
  • Maintainability — 六個月後能改嗎？
  • Hidden risk — 表面看不出的系統性問題
錯誤復盤
Failure Investigator
review fail / production bug 觸發，找系統性問題而非單點錯誤。
  • Root cause analysis — 真正的根因是什麼？
  • Pattern detection — 這是一次性錯誤還是系統問題？
  • Prevention strategy — 如何確保不再發生？
05

Supreme Judge Prompt 結構

輸入包含三個部分：原始任務需求 + 生成結果（完整 code）+ 多個 reviewer 報告（含分數和詳細意見）。Opus 需要完整上下文才能做出有意義的最終判決。

決策規則
⛔ 任何 critical issue valid
→ 強制 REJECT，無例外
⚠ 系統設計缺陷
→ REVISE，附具體修改方向
✓ 生產安全，風險可接受
→ ACCEPT，附信心分數
輸出格式（嚴格 JSON）
{
  "final_verdict": "accept | revise | reject",
  "confidence": 0–1,
  "key_risks": [...],
  "missed_issues": [...],
  "final_reasoning": "..."
}

為什麼嚴格 JSON：Opus 輸出必須機器可讀，才能直接接入 Auto Fix Loop。如果 verdict 是自由文字，下游系統無法自動解析「該不該修」、「修哪裡」。結構化輸出讓整個管線變成真正的自動化閉環，而非半手動流程。

Opus 真正的 強項

不是 Opus 的強項
Opus 不擅長的事
✗ 寫 code（速度太慢、成本太高）
✗ 快速回應（等待時間不適合實時對話）
✗ 低風險 review（殺雞用牛刀，浪費 token）
Opus 的真正強項
✓ 長上下文理解 — 整個 codebase + 多份 review 報告同時消化
✓ 系統層級判斷 — 看到其他模型看不到的架構性問題
✓ 找人類沒看到的風險 — 特別是隱性依賴和安全盲點

Opus 的核心優勢是系統性思考能力：它能在吸收大量上下文後，做出其他模型無法做到的整體性判斷。這恰好是 Supreme Judge 角色最需要的能力。用它來寫第一版 CRUD，是把火箭引擎裝在腳踏車上。

成本最優策略 — 條件觸發

Opus 不應該在每次 review 都出現。精確的觸發條件讓它只在真正需要的時候介入，成本比全量使用降低 80% 以上，效果反而更好。

觸發條件（任一滿足）
◈ 審查綜合分數 < 8 分
◈ Reviewers 之間有衝突意見
◈ 有任何 critical issue 被標記
◈ Production code（非 staging）
◈ Infra / Agent 系統變更
路由邏輯（Python 示意）
def route_to_judge(task, reviews):
  score = aggregate_score(reviews)
  has_conflict = detect_conflict(reviews)
  has_critical = has_critical_issue(reviews)

  if task.is_simple and score >= 8:
    return skip_opus() # 不觸發

  if (score < 8 or has_conflict
      or has_critical or task.is_production):
    return use_opus() # 觸發 L4
成本原則
精準觸發 > 全量使用
80% 的 review 不需要 Opus。剩下 20% 的高風險 case 才是 Opus 真正有價值的戰場。
09

Reviewer Ranking System — 加權判決

長期記錄各模型在不同 review 類型的表現，讓 Opus 不是對所有 reviewer 意見一視同仁，而是按歷史準確率加權做最終裁決。

模型 最強項 相對弱項 Opus 權重方向
Claude Sonnet 邏輯 bug、規格合規、DRY 違反 邊緣案例、UI 視覺問題 邏輯問題高權重
Gemini Edge case、多模態、異常路徑 深層業務邏輯 邊界問題高權重
GPT 穩定、格式一致性、說明清晰 相對保守，漏報率較高 結構問題參考

實作方式：每次 Opus 裁決後，記錄「哪個 reviewer 的意見最終被 Opus 採納」。長期統計形成各模型的 accuracy_by_category 向量，作為 Opus Prompt 的動態上下文，讓 L4 判決越來越精準。

立即行動清單 — 四步到位

1
建立 Supreme Judge（Opus）路由邏輯
實作 route_to_judge() 函數，根據 score / conflict / critical / production 四個條件決定是否觸發 Opus L4。這是整個系統的核心路由決策點。
2
強制 JSON review 結構
讓所有 reviewer（Sonnet、Gemini、GPT）的輸出都符合統一 JSON schema：{ score, issues[], verdict_suggestion, confidence }。Aggregator 和 Opus 才能機器可讀地處理多份報告。
3
做 Auto Fix Loop
Auto Fixer 接收 Opus 的 key_risks 和 missed_issues，做定向修復後重新進入 L2 review。設定最大迴圈次數（建議 3 次），防止無限循環。
4
記錄 reviewer 表現數據
建立 reviewer_performance.jsonl，每次 Opus verdict 後寫入哪個 reviewer 的意見被採納。累積 50+ 筆後開始用加權方式影響 Opus Prompt，讓系統持續自我優化。

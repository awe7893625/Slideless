Part 2

T-TAX 稅務顧問系統升級

2.1 系統現況診斷（Scout 分析）

Scout 完整閱讀 7 個核心檔案後，確認以下問題：

Telegram 格式
標頭「⚖️ 財稅顧問回覆」複述問題、顯示毫秒數（工程師視角）、全 emoji 標頭、硬截斷 600 字 — 回覆充滿機器人感，影響顧問形象。
Persona 一致性
TTAX_IDENTITY（v3 路徑）和 _reason() prompt（舊路徑）不一致，兩套 identity 並存，LLM 收到混亂信號。
RAG 法條引用
_format_context() 沒有傳 ruling_no，LLM 拿不到財政部函釋字號，回答缺少精確法源。
LegalReasoner
推理準則未強制「每個結論必須對應法條」，存在 LLM 跳過引用直接回答的風險。
可觀察性
/ping 只確認 API 連通，無法回報 RAG 向量數量，Rain 無法監控知識庫覆蓋率。

2.2 系統架構（Scout 確認完整呼叫鏈）

Telegram 用戶訊息 → tax_bot.py query_tax() → POST /api/tax/query（Kai 5002） → tax_service.py TaxService.query() → tax_query_router.py（路由分類） → tax_rag_service.py TaxRAGService.query() → QdrantRetriever.hybrid_search()（向量搜尋） → Fallback: LightVectorStore（SQLite 本地） → tax_legal_reasoner.py / tax_accounting_reasoner.py → BaseTaxReasoner._call_llm() → claw-proxy（主，Claude Sonnet/Opus） → codex-proxy（備） → Anthropic API 直連（最後 fallback） → format_reply()（Telegram 格式化）

模型說明：T-TAX 使用 claw-proxy（Claude Sonnet/Opus，依問題複雜度路由），非 Claude Code 直接呼叫。LightVectorStore（SQLite）為 Qdrant 無法連線時的本地 fallback，確保服務不中斷。

移除機器人化的標頭結構、毫秒數顯示、多餘 emoji，改為接近真人顧問的對話語氣。

2
tax_base_reasoner.py — TTAX_IDENTITY 人格改寫

加入「執業超過 15 年」具體設定，明確語氣指引：「直接給結論，不說廢話，語氣像朋友，不確定的事明確說需要函詢，引用法條寫名稱條號即可」。讓 LLM 有一致的人格基礎，不再因 prompt 設計隨機漂移。


同步修復 _format_context() 加入 ruling_no 傳遞，LLM 現在能在回答中正確引用財政部函釋字號（如「財政部 85 台財稅第 850123456 號函」），大幅提升法源完整性。

3
tax_rag_service.py — 舊路徑 identity 同步

將 _reason() 方法中的 prompt 開頭更新為與 TTAX_IDENTITY 一致的人格描述，消除兩條呼叫路徑的 persona 不一致問題。LLM 在任何路徑下收到的都是同一個「T-TAX 顧問」設定。

4
tax_legal_reasoner.py — 推理準則強化

【專業準則】第 3 點加強為：「每個結論必須對應至少一條法條名稱＋條號或函釋字號」，確保 LLM 不跳過引用法條直接給出建議，符合 Rain 確立的財稅研究法源完整性 SOP（mother law → sub-law → ruling number 鏈）。

5
/ping 指令升級 — RAG 可觀察性

新增查詢 /api/tax/migrate/status，/ping 現在回報：

  • Qdrant 向量數量（雲端知識庫）
  • LightVectorStore 向量數量（本地 fallback）
  • RAG 覆蓋率狀態

Rain 現在可以隨時用 /ping 確認知識庫是否有在增長。

2.4 Scout 的額外發現

Scout 在審查中發現 _reason() 和 _select_model() 這兩個方法在 v3 路徑下可能已是 dead code（不再被呼叫）。建議 Quill 在下次 code review 時確認是否可清理，減少維護負擔和 prompt 不一致風險。

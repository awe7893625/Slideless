02 · M1

Real-time Span Instrumentation

每次 LLM call 前後自動插入 span，不阻塞主流程。

1
emit_span_start()

記錄 trace_id, span_id, model, input_summary, start_time 到 _span_registry（in-memory dict）

2
LLM call 執行

llm_service.py 正常呼叫 Claude / Gemini / GPT，不影響任何現有邏輯

3
emit_span_end()

從 registry 取出 span，計算 latency_ms + cost_usd，daemon thread 背景送 LangFuse

4
POST /api/public/ingestion

發送 GENERATION observation，包含 startTime / endTime / usage / metadata

📦 Span 資料欄位

  • span_id — UUID v4
  • trace_id — = task_id（關聯任務）
  • model — 實際呼叫的模型名稱
  • input_summary — 前 300 字元
  • output_summary — 前 300 字元
  • latency_ms — 毫秒精度
  • cost_usd — 依模型費率精算

⚡ 設計亮點

  • 完全 fire-and-forget：daemon thread，失敗不影響任務
  • trace_id 可選參數 — 不傳就不追蹤，向後相容
  • linter 陷阱：import time # noqa: F401 防止 ruff 誤刪
03 · M2

LLM Judge Evaluation Framework

任務完成後，Claude Haiku 自動擔任品質評審官，4-axis 加權評分。

🏆 4-Axis 評分維度

維度說明
Correctness任務是否達成目標？輸出是否準確？
Completeness需求是否完整涵蓋？有無遺漏？
Format輸出格式是否符合任務類型？
EfficiencyToken 用量 + 執行時間是否合理？

💰 評估成本

  • 評估模型：Claude Haiku 4.5
  • 每次評估 ~500 input + 100 output tokens
  • 單次成本：≈ $0.0008
  • 100 筆/月 = $0.08（幾乎免費）
  • 雙源儲存：SQLite task_evaluations + LangFuse scores

Incident Type閾值指派

HIGH_LATENCYp95 > 1800s（30分）max
BUDGET_ALERT7天成本 > 月預算70%klay
LOW_QUALITY_SCORE近20筆平均 < 6.0klay

⏰ 背景監控

  • 每 15 分鐘自動巡檢一次
  • 首次延遲 5 分鐘（等 LangFuse 服務完全啟動）
  • daemon thread，Kai 重啟自動恢復
  • 手動觸發：POST /api/incidents/check

3
24h 去重

查 SQLite incident_log，同類型 24h 只觸發一次

4
create_task() + Discord

建 Kai task + webhook 通知（DISCORD_WEBHOOK_URL）

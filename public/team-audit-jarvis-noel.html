根本原因

Bug #1：session-start.sh 孤兒任務堆積

~/.claude/hooks/session-start.sh 每次啟動 Claude Code 都會建立 "Terminal Session" 任務，設計為建立後立即 claim+done（純 audit trail）。

問題：claim 或 done 的 curl 呼叫偶爾因網路 timeout 靜默失敗（|| true 吞掉錯誤），導致任務永遠留在 pending 狀態。

影響：12 個 "Terminal Session" 孤兒任務堆積在「待處理」欄位，造成看板視覺混亂。

Bug #2：Stale Executing 任務（Ray 審查）

spawn-ray.py 建立的 Ray 審查任務，對應的 Claude Code session 已結束但未呼叫 /done endpoint。

問題：任務卡在 executing，需等 30 分鐘 timeout 才自動 fail。

影響：看板「執行中」欄位顯示已結束的任務。

前端邏輯驗證

✅ 前端邏輯正確
  • claimed 狀態正確映射到「執行中」欄位
  • 四欄（pending / executing / success / failed）涵蓋所有有效狀態
  • 後端 timeout scanner 每 60 秒執行一次

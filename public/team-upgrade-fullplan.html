4 Blockers · Ray 認定

必須先解決，才能開工

以下 4 項是物理性或安全性前提條件，不解決就開工等於在沙上建房子。

B1
Cloudflare Tunnel — n8n 雲端打不到 Kai 本機
n8n 部署在 Zeabur（雲端），但 Kai 在 127.0.0.1:5002（本機）。沒有 Tunnel，n8n 就無法呼叫 Kai API，整個企劃的核心串接點失效。
✅ 解法：Forge 用 cloudflared 建 Tunnel，綁定子域名（如 kai-internal.yourdomain.com），加 CF Access Policy 限定只有 n8n IP 可通。
B2
Kai API 加 Token 驗證 + automation_mode Flag
Kai 目前無 API 認證，n8n 自動呼叫和人工呼叫無法區分，遷移期間容易重複執行任務（同一個日報發兩次）。
✅ 解法：Max 在 Kai 加 Bearer Token 驗證（一行 FastAPI middleware），加 `source: "n8n"` 欄位區分自動/手動，任務去重邏輯。
B3
n8n Webhook Header Auth — 公開 URL 無驗證
Zeabur 的 n8n Webhook URL 是公開的，任何人都能打，等於任何人都能觸發爬蟲或發文。
✅ 解法：Forge 在每個 Webhook 節點設定 Header Auth（`X-Webhook-Token: secret`），Knox 入 vault 管理。
B4
R2 Token 最小權限 — Account 級 Token 危險
用 Account-level API Token 給 n8n，等於給了 n8n 刪除所有 Cloudflare 資源的能力。一旦洩露後果極嚴重。
✅ 解法：Forge 建立 R2 專屬 API Token，只給 Object Read/Write 權限，範圍限定到單一 bucket。Knox 登記入 vault。

風險評估 · Ray Full Review

7 個風險，逐一拆解

R1-R4 是 Blocker 或高嚴重度，R5-R7 可接受，執行中注意。

#風險嚴重度狀態緩解方案
R1 n8n 單點故障（Zeabur 掛了 → 自動化停擺） Medium 可接受 n8n 是新增層，掛了降回手動模式，核心系統（Kai/OpenClaw）不受影響。穩定 1 個月後考慮 HA。
R2 Webhook URL 安全（可被惡意觸發） Blocker B3 已有解法 Header Auth Token + Knox 管理，每個 Webhook 強制驗證。
R3 R2 API Key 洩露（可刪所有儲存） Blocker B4 已有解法 最小權限 Token（Object Read/Write only）+ Knox vault，不用 Account-level Token。
R4 n8n → Kai 本機串接（網路不通） Blocker B1+B2 已有解法 Cloudflare Tunnel + Kai Bearer Token 驗證，雙重保障。
R5 爬蟲排程過頻被封 IP Medium 注意執行 Cron 間隔 ≥ 4 小時，Spider/Lynx 加 delay 和 User-Agent 輪換，Nautilus 處理高難度目標。
R6 Workflow vs Agent 邊界不清 Low 已明確 原則：確定性邏輯（排程/通知/存檔）→ n8n Workflow；需要 AI 判斷（文案/分析）→ Agent。
R7 遷移期兩套系統並行重複執行 Medium 注意執行 Kai automation_mode flag 區分來源，每個任務加 idempotency key，舊手動流程在對應 n8n workflow 上線後才關閉。

Rain 需要確認的決策

3 個需要你拍板的問題

以下 3 個決策影響執行方向，Ray 已給出建議，但最終由 Rain 確認。

❓ CF Tunnel 用哪個域名？

Cloudflare Tunnel 需要綁一個你有控制權的域名（如 kai-internal.yourdomain.com）。建議用現有 Cloudflare DNS 管理的域名，不對外公開這個子域名。

用現有域名的子域（如 internal.xxx.com），Ray 確認可行
❓ GA4 MCP Server 要不要做？

建 FastMCP GA4 Server 可以讓 Atlas 直接用 MCP tool 查 GA4 數據，不用每次手動跑 API。Darrell 因怕商業帳號洩漏沒開源，我們內部用 localhost 綁定完全安全。

做，內部用安全，Atlas 效率大幅提升
❓ GooayeAI Pipeline 納入本次遷移？

GooayeAI 已有穩定的 RSS → Gemini → Telegram pipeline，目前只是偶爾需要手動重啟。遷移到 n8n 有一定風險，且效益不如其他項目高。

不急，下一個 iteration 再做，先穩核心

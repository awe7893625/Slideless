Overview

目前團隊到底怎麼站著運作

現在的 Rain AI Team 是董事會策略層、Klay 總控、HQ 協調、部門 owner、review spine、外部 specialist 的混合系統。核心不是把所有角色塞進同一模型，而是每個角色有自己的 primary model、fallback class、責任邊界與升級規則。

read this first
手冊已改成固定網址 與團隊相關的重大更新優先回寫這一頁；只有一次性的審計、戰役、改版報告，才另外做新頁並進 archive。
現在的 canonical 不是舊版 Haiku-heavy 2026-03-24 之後，核心工程、研究、安全、審查與部分投研資料角色已回到 Claude Sonnet 4.6，Haiku 保留給協調、文書與模板化分析席位。
Codex contingency 已分級 Opus-tier 切 gpt-5.4，Sonnet-tier 切 gpt-5.3-codex，Haiku-tier 與 specialist contingency 切 gpt-5.4-mini，Ray 保持 codex-first。

02
Board Layer

董事會是方向決策層，不是 execution 替代品

董事會只在 strategic choice 出現時介入，例如產品方向、資源配置、商業模式、新市場與高風險專案。純技術 hotfix、production incident、payment incident 直接走 Klay 的執行鏈，不需要董事會兜一圈。

Trigger: Critical + strategy choice

何時應啟動董事會

新產品方向與是否進入新市場。
重大資源配置與是否重組部門。
高風險高不確定性的專案是否啟動。
Rain 明確下達 /board [topic]。

3Ray / Vick / Gemini Pro advisory
→
4策略裁決
→
5回到工程三階段執行

Model Stack

模型堆疊要分成 primary roster 和 contingency lane 看

Operating System

你下指令後，任務理論上應該怎麼走

這套團隊應該像 operating system 一樣運作，而不是單一聊天視窗。Rain 直接對 Klay 下任務，Klay 先 intake、classify、route，再把工作交給 owner 執行；OM 只做作戰室級別的協調；Quill、Forge、Ray 組成 review spine。

intake → classify → route → execute → review

任務流程圖

標準路徑應先分級，再決定是否 phase 拆分與多人協作。

1Rain task → Klay intake
2Classify → route → assign owner(s)
3Execute + parallel collaboration
4Quill / Forge / Ray review → acceptance-ready
Tiny / Standard / Major / Critical Tiny 可以直接完成；Standard 至少有 owner 加 review；Major/Critical 要有 phase、ownership、驗證與剩餘風險。
Review spine Quill 看 code quality 與 diff，Forge 看 build/test/deploy gate，Ray 看方案與設計風險。三者不是互斥，而是不同角度的防線。
特殊 domain 規則 Website 強制先出規格包；Investment Phase 1 report-only；Research 逆向分析走 PRES；Finance 由 Inquisitor 發現、Knox 決策與執行。

claw-claude-proxy/src/team-models.js

同步鏈條

這頁不是原始資料庫，而是整理後的手冊。所以來源必須有先後順序，不然 roster、proxy、memory 很容易再次分叉。

Rules layer `CLAUDE.md` / `team-roster.md` 定義 command 與 board 規則。
Canonical layer `team.json` 定義 roster 與 primary model，是手冊與系統對齊的主鍵。
Runtime layer `team-models.js`、`team_service.py`、`agent_task_service.py` 讓 proxy / Kai 依同一口徑運行。
Narrative layer `MEMORY.md`、`team-manual.html`、`team-reports.html` 只負責人讀版本，不反向覆蓋 canonical。

衝突時怎麼判

這裡把判斷規則寫死，避免看到舊報告時還要重新猜哪份才是對的。

Rule A 模型衝突

先信 `team.json`，再回頭修 proxy 與手冊，不反過來。

Rule B 組織衝突

先信 `team-roster.md` 的 command 結構，再檢查手冊是否漏更新。

Rule C 歷史報告衝突

歷史頁只保留當時上下文，不能覆蓋現行手冊。

Rule D Runtime 異常

看 `team-models.js` 與 Kai / proxy 實際輸出，不靠頁面文案猜狀態。

本頁刻意修正的舊口徑 舊報告曾把 Market Intel / News Collector 寫成 Haiku，也把 Knox 長期寫成固定 Opus。這份手冊已用目前 canonical 修正。
頁面用途 這一頁只負責「現在有效的版本」。任何一次性 upgrade report、設計 review、審計紀錄，改登錄到 archive，不再把手冊寫成月報格式。

審查背景

Situation — 現況

Rain AI Team 目前有 兩條任務下達路徑並行運作：Rain 透過 Noel Telegram 遠端派發（異步），以及在 Klay Claude Code 互動 Session 直接下指令（同步）。兩條路徑共用 Kai 看板作為任務追蹤中樞，但執行機制截然不同。

Complication — 問題

隨著專案規模增長（5+ 活躍專案、19 位 Agent），兩條路徑的能力落差逐漸擴大。Telegram 路徑無法執行 3-Phase Planning、無法並行 dispatch、且缺乏進度回報機制。安全防護水準也不一致（spawn-ray.py 落後 spawn-klay.py 至少 4 項防護）。

Question — 核心問題

兩條路徑各有什麼盲點、斷點、延遲點？如何在不增加 Rain 操作負擔的前提下，提升整體執行效率與可靠性？

Answer — 結論

共識別出 8 項盲點，其中 4 項為 P0/P1 需立即修復。最關鍵的是：Telegram 路徑缺乏超時機制（殭屍任務風險）、spawn-ray.py 安全防護不足、以及完成通知缺失。建議按 4 階段路線圖逐步強化。

盲點 & 斷點分析

路徑 A 盲點（Noel Telegram）

P0 Critical

B1：殭屍任務 — 無超時機制

spawn-klay.py 啟動 tmux 後，若 Claude 卡住或靜默失敗，Kai 看板任務永遠停留在 "executing"。沒有 watchdog、沒有超時中斷。長期累積會造成看板失真，Rain 無法判斷真實工作負載。

P0 Critical

B2：spawn-ray.py 安全防護不足

相比 spawn-klay.py 缺少 5 項防護：無危險指令偵測、無 Prompt Injection 消毒、無私有目錄權限、無早期失敗偵測、無失敗回報。Ray 處理的是 code review 任務，可能接收到含攻擊性內容的程式碼片段，風險不低於 Klay。

P1 Medium

B3：完成通知黑洞

任務完成後僅更新 Kai 看板 + openclaw event（可能離線）。Rain 除非主動查看 Kai Dashboard 或 tmux log，否則不知道任務已完成。需要主動推播到 Discord。

P1 Medium

B4：無法觸發 3-Phase Planning

spawn-klay.py 產生的 prompt 只告訴 Klay「你是 Klay，執行這個任務」，沒有明確指示走 3-Phase Protocol。Klay 在 tmux 中雖然可以使用 Agent tool，但 CLAUDE.md 中的 3-Phase 規範是否被正確載入取決於 workdir 的 .claude 配置。

P2 Low

B5：重複偵測過度簡化

以任務標題精確比對作為去重依據。「重構購物車」和「購物車重構」會被視為不同任務。需改為模糊比對或加入任務指紋（hash of description）。

路徑 B 盲點（Klay 互動 Session）

P1 Medium

B6：Session 中斷 = 工作遺失

若 Claude Code 崩潰或 Rain 意外關閉終端，所有進行中的 subagent 工作會中斷。雖然有 handoff 機制，但依賴 Session 正常結束時才觸發。異常關閉時不會自動 handoff。

P2 Low

B7：單一 Session 瓶頸

Rain 同時只能有一個 Claude Code 互動 Session。若正在執行大型任務，新的緊急任務只能等待或走 Telegram 路徑。兩條路徑之間沒有任務佇列做負載均衡。

P3 Low

B8：Kai 看板同步斷點

路徑 A 由 spawn script 自動建立 Kai 任務，但路徑 B 依賴 Klay 手動呼叫 API。兩條路徑的 task_type、priority 格式不完全一致，可能造成看板統計失真。

改善建議

P0 — 立即修復

P0R1：spawn-ray.py 安全強化

從 spawn-klay.py 移植所有安全機制：危險指令偵測、sanitize_for_prompt、私有目錄 700 權限、早期失敗偵測、fail_kai_task 回報。

effort: 1h | owner: Max | files: spawn-ray.py

P0R2：任務超時 Watchdog

新增 watchdog 腳本，每 10 分鐘掃描 Kai 看板 "executing" 狀態的任務。超過 2 小時且 tmux session 已結束者 → 自動標記 fail + Discord 通知。

effort: 2h | owner: Forge | files: watchdog script + LaunchAgent

P1 — 本週內

P1R3：完成通知推播

在 spawn prompt 的完成指令中加入 Discord webhook 呼叫。任務完成時同步推播到 #task-complete 頻道，包含摘要和耗時。

effort: 1h | owner: Noel | files: spawn-klay.py, spawn-ray.py

P1R4：Spawn Prompt 加載 3-Phase

在 build_prompt 中加入明確指示：若任務符合 Major/Critical 條件，必須執行 3-Phase Planning Protocol。確保 workdir 設定正確以載入 CLAUDE.md。

effort: 1h | owner: Max | files: spawn-klay.py

P2 — 兩週內

P2R5：Crash-safe Handoff

Klay 互動 Session 每完成一個子任務就寫入 checkpoint 到 handoff 檔案。即使 Session 異常中斷，下次啟動時可從最近的 checkpoint 恢復。

effort: 3h | owner: Klay | files: session-protocol, CLAUDE.md

P2R6：模糊重複偵測

將 check_duplicate 改為：標題 Levenshtein distance < 5 或 description hash 相同即視為重複。防止同任務不同措辭被重複執行。

effort: 1h | owner: Max | files: spawn-klay.py, spawn-ray.py

P3 — 下月

P3R7：統一任務佇列

在 Kai 中建立 priority queue，兩條路徑統一匯入。支援任務優先級排序、自動分配到可用 Agent、避免單路徑瓶頸。

effort: 8h | owner: Max + Forge | files: Kai backend

P3R8：Kai 看板欄位標準化

統一兩條路徑的 task_type、priority、source 欄位格式。加入 path_origin 欄位（telegram / claude_code）方便追蹤。

effort: 2h | owner: Rex | files: Kai task API

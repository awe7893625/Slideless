FAP 的 API 直發架構（最有價值的發現）

FAP 完全不操作 Facebook DOM，而是用 declarativeNetRequest 偽造 Origin/Referer headers，直接 POST 到 Facebook 內部 GraphQL endpoint。圖片走 upload.facebook.com API 上傳。

mbasic → 提取 fb_dtsg token + c_user cookie
  ↓
declarativeNetRequest 偽造 headers
  ↓
GraphQL POST → 發文成功（不碰任何 DOM）
  ↓
upload.facebook.com → 圖片上傳 → photo_id → 附到發文

Phase 2 考慮採用此路線解決圖片支援問題。

Phase 1 已交付（v1.3.0）

Done Spintax 引擎

支援巢狀語法 {你好|嗨|哈囉}，每個社團收到不同版本的文案。內建 20 層深度安全限制，防止無限遞迴。

Done 智慧延遲系統

根據目標社團數量自動推薦延遲範圍（60s~1200s），加 ±20% 隨機抖動。超過 50 個社團自動建議 5 分鐘以上間隔。

Done 重試機制

發文失敗自動重試，每次間隔 5-8 秒（隨機），最多 3 次嘗試。每次重試重新開 tab 避免狀態汙染。

Done 3-Slot 內容輪替

最多 3 組不同文案，round-robin 輪替發送。搭配 Spintax，理論上可產生無限種文案組合。

Done 即時進度顯示

成功/失敗即時計數 + 進度條 + 下一個社團倒數計時。完成後彈出統計摘要（成功率、總耗時）。即使 popup 關閉，進度也存在 storage 中，重開可恢復。

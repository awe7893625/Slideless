Section 01

執行摘要（SCQA）

Situation → Complication → Question → Answer
Situation · 現況
全球前 5%

Rain AI Team 目前擁有 28 個專職 agent、三階段規劃流程、多層記憶系統，與夜班自主執行能力。以個人/小團隊標準衡量，已進入全球前 5%。

Complication · 問題
人工協調成本指數成長

系統核心仍依賴 Markdown 文件 + Klay 人工理解驅動流程，無狀態機、無 call-level tracing、無自動錯誤恢復。系統規模越大，協調成本呈指數成長。

Question · 核心問題
如何升級而不破壞優勢？

如何在不破壞現有規模優勢的前提下，引入企業級基礎設施——狀態機、可觀測性、自動恢復——並按 ROI 排序推進？

Answer：按 ROI 排序，優先推進三條路線： B（Observability）→ A（圖引擎）→ C（雲端化）。 最小有效動作是 30 天內將 LangFuse 接入 Kai 中控台，成本 2.5 週，讓整個系統從「看不見」變成「可被觀測」。

Section 05

架構簽核

Ray 審查結論 · 風險 · 啟動順序
⚠
審查結論：有條件通過 — 三大技術風險需於開工前確認方案
1
DAG 引擎的 CLI 整合問題

Claude Code CLI 無原生 callback，DAG 需 polling Kai API，最長可能產生 30 秒延遲。必須先設計 signal 機制（webhook 或輪詢間隔策略），才能開工路線 A。

2
Observability 的 Prompt 完整性限制

LangFuse 只能記錄任務描述，非完整 prompt（Claude Code CLI 不暴露原始 prompt）。需每個 subagent 回傳結構化的 prompt_summary 欄位，才能達到有效 tracing。

3
Schema 向後相容性要求

任何 Kai schema 變更必須向後相容 30 天，舊欄位不得直接刪除。新增欄位（agent_steps、depends_on）需設計 migration 版本與 nullable 策略，避免破壞既有夜班任務查詢。

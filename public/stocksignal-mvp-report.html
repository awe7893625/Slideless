2決策：跳過自訂域名（本次）

背景

原計畫透過 api.stockcast.tw 提供後端服務，需將 Dynadot 的 NS 修改至 Cloudflare。

遭遇阻礙

Dynadot 登入頁啟用了 Cloudflare Turnstile 機器人驗證，Playwright 自動化及 Chrome DevTools 均無法通過，顯示「Verification failed」。自動化登入路徑在此受阻。

Rain 決策

跳過自訂域名，改用 Cloudflare Quick Tunnel，加速上線時程。 後續 Dynadot NS 修改由 Rain 親自在瀏覽器操作（繞過機器人驗證），再啟用 api.stockcast.tw。

3Quick Tunnel 自動化機制

啟動腳本位置

/Users/rain/Projects/StockSignal/start-tunnel.sh

腳本執行流程

  • 啟動 cloudflared 快速 tunnel
  • 等待並抓取新的 tunnel URL
  • 自動更新 .env.local
  • 自動更新 Vercel 環境變數
  • 自動觸發 Vercel production redeploy

為何需要此機制

  • Quick Tunnel URL 每次啟動都會變更
  • 前端需要正確的後端 URL 才能運作
  • 腳本全自動化處理更新流程
  • 無需人工干預即可完成重啟
# start-tunnel.sh 核心流程 cloudflared tunnel --url http://localhost:8001 & # 啟動 tunnel # 等待 URL 輸出，正則抓取 *.trycloudflare.com TUNNEL_URL=$(grep -o 'https://[^.]*\.trycloudflare\.com') # 更新 .env.local echo "NEXT_PUBLIC_API_URL=$TUNNEL_URL" > .env.local # 更新 Vercel 並重新部署 vercel env add NEXT_PUBLIC_API_URL && vercel --prod --yes
